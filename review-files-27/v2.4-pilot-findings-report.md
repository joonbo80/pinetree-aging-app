## Findings — answers to spec draft open questions

### Q1. Are weekly statements re-issues or deltas?

**ERP data answer: DELTA-only.**

Evidence:
- INVOICE internalId: baseline 16822-17359 → May 17360-17377. **0 overlap.**
- AP internalId: baseline 29398-29944 → May 29945-29989. **0 overlap.**
- INVOICE business ID (PEIN sequence): baseline ...19479 → May 19480-19500. **0 overlap.**
- Date ranges: baseline 01-01 to 04-30, May 05-01 to 05-05. **0 overlap.**

Every weekly cut contains ONLY new transactions issued since the last upload. The ERP assigns sequential internalIds, so byte-identical re-uploads are easily detected.

**Statement answer: still unknown.** Pilot did not include LOCAL/AGENT statement files. Statement re-issue vs delta behavior is the remaining open question.

### Q2. Are AP costLineCode patterns standardized?

**No.** The costLineDesc field is free-text:
```
TRUCKING COST
TRUCKING CHARGE COST
DETENTION CHARGE COV
DETENTION CHARGE 1 D
ISF FILING FEE
TERMINAL CHARGE COST
AIR FREIGHT CHARGE M
EDI CORRECTION
OCEAN FREIGHT US COS
CN STORAGE 2 DAYS
AIRLINE TRANSFER FEE
```

Implication: match key uses costLineDesc + amount + currency. Free-text is fine as long as it's reproducible across uploads of the same row.

### Q3. PO# / BL# / invoiceNo / ourRefNo frequency in real data?

| Field | Source | Present % |
|------|--------|-----------|
| invoiceNo | INVOICE | 100% (PEIN sequence) |
| ourRefNo | INVOICE | ~76% (RSTOR/HJTS/HYVR/etc.) |
| ourRefNo | AP | ~95% |
| vendorInternalId | AP | 100% |
| vendorName | AP | ~98% |
| BL# | none | 0% (not observed in pilot) |
| PO# | none | 0% (not observed in pilot — `orderRef` field is internal PEOI/PEAI sequence, not customer PO) |

**Implication**: BL# and PO# are out of scope for v2.4. Match keys use invoiceNo / ourRefNo / vendorInternalId only.

### Q4. Batch volume per year?

```
5 business days (May 1-5):
  INVOICE  17 rows
  CRDR      7 rows
  AP       41 rows
  TOTAL    65 rows
```

Annual estimate: 52 weekly batches × 65 rows ≈ 3,400 ERP rows/year. IndexedDB handles this with 100x headroom. Storage sizing is a non-issue.

### Q5. Batch-level vs effective export?

Deferred — depends on statement weekly behavior, which we don't have data for yet.

## CRITICAL NEW FINDING #1 — AP multi-cost-line CONFIRMED with hard data

User's reviewer-flagged concern is fully validated:

```
ourRefNo "WSATOR261111" → 3 cost lines:
  WIN YAN LOGISTICS    -165.0 USD  DETENTION CHARGE COV   (vInv 29989)
  HAPAG LLOYD CANADA    165.0 USD  DETENTION CHARGE 1 D   (vInv 29967)
  WIN YAN LOGISTICS     900.0 CAD  TRUCKING COST          (vInv 29958)
```

**40 AP rows in the pilot → 7 shipments (17.5%) have 2+ cost lines.**

If v2.4 dedup uses `ourRefNo + amount + currency` alone, it would silently drop the cost lines that happen to share an amount, even though they're legitimately distinct (different vendor + different cost type). This is the silent-corruption class.

**Resolved**: AP_LINE match key MUST include `vendorName + costLineDesc` in addition to `amount + currency`. Documented in v2.4-spec-draft. Verified by data.

## CRITICAL NEW FINDING #2 — "CRDR" file is NOT one source type

This was unexpected. The baseline `2_CRDR_JAN-APR_2026.xls` and the pilot `2-1_CRDR_MAY_1-5_2026.xls` files share the same filename pattern but contain **completely different data**:

```
BASELINE (Jan-Apr CRDR):
  - 19 columns, 346 rows
  - Business ID col 1 = "PECDR016797" (CR/DR memo numbers)
  - Vendor col 2 = "SINO LOGISTICS (MALAYSIA)"
  - Has invoice-style aging columns (post date, due date, amounts)
  - This is CR/DR memos issued — accounting documents

PILOT (May 1-5 "CRDR"):
  - 15 columns, 7 rows
  - Business ID col 1 = "EFT" / "E-TRANSFER" (payment method!)
  - Party col 2 = "PANTOS LOGISTICS CANADA INC."
  - Has bank/cheque columns ("HANA KOREA EXCHANGE BANK", "CHEQUE ISSUED : RBC 5431")
  - This is PAYMENT ENTRIES — transactions, not memos
```

**These are two different domain entities sharing one filename pattern.**

The baseline CRDR file = "credit/debit memos" (a kind of accounting document).
The May "CRDR" file = "cash receipts / disbursements" (payment register entries).

Both might legitimately be called "CRDR" colloquially, but they have different schemas, different keys, and different match rules.

### Implications for v2.4 spec

1. **Parser cannot trust filename alone** — must inspect column structure to determine actual source type.
2. **Source type taxonomy needs to expand**:
   - `INVOICE` (existing)
   - `CRDR_MEMO` (was just `CRDR` in v2.0-2.3 — the baseline kind)
   - `PAYMENT_REGISTER` (NEW — the May kind, EFT/cheque records)
   - `AP` (existing)
3. **v2.4 step v2.4-3 (match key construction) must add PAYMENT_REGISTER_LINE** as a new match-key variant.
4. **Backward compat**: v2.0-2.3 baseline JSON has 346 "CRDR" rows that are actually CRDR_MEMO. A migration step must re-tag them.

### Why this matters for Phase 3

If v2.4 ships with single `CRDR` source type and SharePoint adapter is built on that, Phase 3 will require schema migration in production. Catching this NOW saves that pain.

### Question for user

Is the May "CRDR" file actually meant to be a separate source type, or is it the SAME ledger but a different report view that the ERP exports? Need to confirm before v2.4 spec freeze.

## Cross-source linking confirmed

`ourRefNo` is the natural shipment identifier across INVOICE / AP. In the May pilot:
- 12 distinct ourRefNos in INVOICE
- 27 distinct ourRefNos in AP
- 8 ourRefNos appear in BOTH → cross-source links

Sample shared refs:
```
RSTOR2603021    → INVOICE 870 CAD + AP CN-ACC 0 CAD + AP WIN YAN 725 CAD
RSTOR2603022    → INVOICE 1620 CAD + AP CN-ACC 750 CAD + AP WIN YAN 725 CAD
RSTOR2603110    → INVOICE 1045 CAD + AP WIN YAN 900 CAD
919JSELG32447900 → INVOICE 1045 CAD + AP WIN YAN 900 CAD
KR17501865       → INVOICE 660 USD + AP SWIRE 660 USD
```

This is the expected pattern: a shipment generates one INVOICE (revenue side) and 1+ AP entries (cost side). v2.4 should preserve this linking — Party Detail already does it via the existing `ourRefNo` field, no change needed for that aspect.

## Methodology note

This pilot accomplished what spec freezing on assumptions cannot: it surfaced 2 critical findings (AP multi-line proof, CRDR-is-two-types) that would have caused expensive rework in v2.4 implementation. The CRDR finding alone is worth the entire pilot session — without real data, no amount of spec discussion would have caught it.

For Phase 3 SharePoint design, the same pattern should apply: pilot with a small SharePoint List and 1 week of synthetic batches BEFORE locking the schema, to surface unexpected domain rules.
