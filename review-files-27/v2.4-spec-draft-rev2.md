## North Star

The app shifts from "single snapshot analyzer" to "time-series accumulator."
Weekly/monthly/quarterly/year-end uploads append to a persistent store,
deduplicated by reference number with explicit user decisions, with full
audit history. This is the foundation Phase 3 SharePoint persistence
will sit on.

## Core concepts

### Import Batch

Every Excel upload creates ONE Import Batch. The batch is the unit of
de-duplication, audit, and rollback.

```typescript
interface ImportBatch {
  batchId: string;              // UUID
  uploadedAt: string;           // ISO 8601
  uploadedByUserId?: string;    // Phase 3 only — null in v2.4
  parserVersion: string;        // e.g. "1.3.0"
  schemaVersion: string;        // e.g. "1.0"

  // Period this batch covers
  period: PeriodKey;
  cycle: StatementCycle;

  // Source files in this batch
  sourceFiles: Array<{
    fileName: string;
    fileSize: number;
    sourceType: 'INVOICE' | 'CRDR' | 'AP' | 'AGENT_STATEMENT' | 'LOCAL_STATEMENT_USD' | 'LOCAL_STATEMENT_CAD';
    rowCount: number;
    sha256: string;             // for byte-identical re-upload detection
  }>;

  // Dedup decisions made when this batch was imported
  dedupDecisions: DedupDecision[];

  // Lifecycle
  status: 'draft' | 'committed' | 'reverted';
  committedAt?: string;
  revertedAt?: string;
  revertReason?: string;
}
```

### Period Key (ISO 8601 compatible)

```typescript
type PeriodKey =
  | `${number}-W${number}`        // weekly:    "2026-W18"
  | `${number}-${number}`         // monthly:   "2026-04"
  | `${number}-Q${1|2|3|4}`       // quarterly: "2026-Q2"
  | `${number}-12-FY`             // yearend:   "2026-12-FY"
  | `${number}-${number}-${number}-AH`;  // adhoc:  "2026-05-15-AH"

type StatementCycle = 'weekly' | 'monthly' | 'quarterly' | 'yearend' | 'adhoc';
```

All formats sort lexicographically AND chronologically. Parseable. Human-readable.

### Statement Period (separate from PeriodKey)

The statement itself covers a date range, which may not equal the period
key (e.g. an ad-hoc upload on 2026-05-15 might cover 2026-W18 statements).

```typescript
interface StatementPeriod {
  statementPeriodFrom: string;  // ISO date
  statementPeriodTo: string;    // ISO date
  statementDate: string;        // ISO date — when statement was issued
  cycle: StatementCycle;
}
```

### Dedup Decision

When a new batch is uploaded, the system finds candidate duplicates from
prior batches and asks the user. Decisions are RECORDED, not silently applied.

```typescript
interface DedupDecision {
  decisionId: string;
  newBatchId: string;
  newRowKey: string;            // {sourceFile}|{sourceRow}
  candidateMatches: Array<{
    priorBatchId: string;
    priorRowKey: string;
    matchKey: MatchKey;          // see below
    similarity: 'exact' | 'amount-mismatch' | 'date-shift' | 'partial';
  }>;
  decision: 'keep_both' | 'exclude_new' | 'replace_prior' | 'merge_link';
  decidedAt: string;
  decidedByUserId?: string;     // Phase 3 only
  reason?: string;              // optional user note
}
```

### Match Key — source-specific (CRITICAL — PILOT-CONFIRMED)

This is where v2.5 prior draft (single global rule) was wrong. Per
reviewer note: AP has multi-line cost structure that requires special
handling.

**PILOT-CONFIRMED**: 17.5% of pilot AP shipments have 2+ cost lines
sharing the same `ourRefNo`. Sample:
```
ourRefNo "WSATOR261111":
  WIN YAN     -165 USD  DETENTION CHARGE COV  vInv 29989
  HAPAG LLOYD  165 USD  DETENTION CHARGE 1 D  vInv 29967
  WIN YAN      900 CAD  TRUCKING COST         vInv 29958
```

**PILOT-CONFIRMED**: BL# and PO# fields are NOT present in pilot data.
Removed from match key variants.

**PILOT-NEW**: PAYMENT_REGISTER source type added (see Source Type
Taxonomy section below).

```typescript
type MatchKey =
  | { type: 'INVOICE_LINE';
      invoiceNo: string;            // PEIN... — observed in 100% of pilot rows
      amount: number;
      currency: 'USD'|'CAD';
      party: string; }

  | { type: 'CRDR_MEMO_LINE';        // renamed from CRDR_LINE for clarity
      crdrMemoNo: string;            // PECDR... — credit/debit memo identifier
      amount: number;
      currency: 'USD'|'CAD';
      party: string; }

  | { type: 'PAYMENT_REGISTER_LINE'; // NEW — pilot finding #2
      paymentMethod: string;         // EFT, E-TRANSFER, CHEQUE
      vendorInternalId: number;      // ERP-assigned sequential ID
      amount: number;
      currency: 'USD'|'CAD';
      party: string; }

  | { type: 'AP_LINE';
      vendorInternalId: number;      // ERP-assigned, 100% present in pilot
      ourRefNo: string;              // shipment-level identifier, 95% present
      vendorName: string;            // CRITICAL — must include for multi-line
      costLineDesc: string;          // CRITICAL — free-text but reproducible
      amount: number;
      currency: 'USD'|'CAD'; }

  | { type: 'STMT_LINE';             // PILOT-OPEN — no statement data tested yet
      statementRef: string;
      amount: number;
      currency: 'USD'|'CAD'; }

  | { type: 'AMBIGUOUS';
      reason: string;
      fallbackFields: Record<string, string>; };
```

CRITICAL DOMAIN RULE: AP_LINE matching uses vendorName + costLineDesc.
Two rows with the same ourRefNo + amount but different vendor or cost
line are DISTINCT (e.g. "WIN YAN trucking 900" vs "HAPAG LLOYD detention 165")
and must NOT be auto-deduplicated. This rule is enforced at the
match-key construction layer, not left to the dedup engine.

### Source Type Taxonomy (PILOT-NEW)

The pilot revealed that "CRDR" filename pattern actually maps to TWO
distinct source types in real data:

| File pattern | Real source | Schema | Detection rule |
|--------------|------------|--------|----------------|
| `*INVOICE*.xls` | INVOICE | 26 cols, business ID = "PEIN..." | filename + col count |
| `*CRDR*.xls` baseline | CRDR_MEMO | 19 cols, business ID = "PECDR..." | col count = 19 AND col[1] starts with "PECDR" |
| `*CRDR*.xls` weekly  | PAYMENT_REGISTER | 15 cols, col[1] = "EFT" or "E-TRANSFER" or "CHEQUE" | col count = 15 AND col[1] in payment method enum |
| `*AP*.xls` | AP | 25 cols, col[1] = vendor name | filename + col count |
| `*AGENT*STATEMENT*.xls` | AGENT_STATEMENT | varies | filename pattern |
| `*LOCAL*STATEMENT*USD*.xls` | LOCAL_STATEMENT_USD | varies | filename pattern |
| `*LOCAL*STATEMENT*CAD*.xls` | LOCAL_STATEMENT_CAD | varies | filename pattern |

The parser must inspect column count + first few cell values to
disambiguate when filename is ambiguous. **Filename alone is NOT enough.**

This is a v2.4 hard requirement, not a v2.5 nice-to-have:
- Without this disambiguation, the May `2-1_CRDR_MAY_1-5_2026.xls` file
  loaded by today's parser would be misinterpreted (wrong column
  positions for amount, currency, ID).
- A migration step is required for v2.0-2.3 baseline JSON: re-tag the
  346 existing "CRDR" rows as "CRDR_MEMO" since they were all that type.

### As-of Date (aging multi-cadence)

```typescript
interface AgingQuery {
  asOfDate: string;             // ISO date — when aging is calculated
  partyKey?: string;             // optional filter
  source?: 'INVOICE' | 'CRDR' | 'AP';
}

// Result is signed-aging at that point in time across all committed batches
function selectAging(query: AgingQuery, store: TransactionStore): AgingResult;
```

Period-end presets (UI convenience):
- "End of last quarter" → asOfDate = quarter-end ISO date
- "Year-end 2025"       → asOfDate = "2025-12-31"
- "Last weekly close"   → asOfDate = most recent Friday or week-end


## Storage layer (adapter pattern)

```typescript
interface BatchStore {
  // Writes
  saveBatchDraft(batch: ImportBatch, transactions: PreviewTransaction[]): Promise<string>;
  recordDedupDecision(batchId: string, decision: DedupDecision): Promise<void>;
  commitBatch(batchId: string): Promise<void>;
  revertBatch(batchId: string, reason: string): Promise<void>;

  // Reads
  listBatches(filter: { cycle?: StatementCycle; from?: string; to?: string; status?: BatchStatus }): Promise<BatchSummary[]>;
  loadBatch(batchId: string): Promise<{ batch: ImportBatch; transactions: PreviewTransaction[] }>;
  findCandidateMatches(newRow: PreviewTransaction): Promise<DedupCandidate[]>;
  getEffectiveTransactions(asOfDate: string): Promise<PreviewTransaction[]>;
  // ^ resolves dedup decisions across all committed batches
}
```

v2.4 implementation: `IndexedDbBatchStore` — works in-browser, single user.

Phase 3 implementation: `SharePointBatchStore` — same interface, talks to
SharePoint Lists. Migration is `new SharePointBatchStore()` swap.

This is NOT premature abstraction — SharePoint is a confirmed future
target, so the seam is required from v2.4 day one.

## Open questions — status after pilot

### RESOLVED by pilot

1. **Are weekly statements RE-ISSUES of prior data, or DELTAS?**
   **PILOT-CONFIRMED for ERP**: DELTA-only. internalId is sequential and
   never overlaps across weekly cuts.
   - Default decision: `keep_both` (every row is naturally new)
   - Re-upload detection: sha256 of file + internalId range overlap

2. **Does AP costLineDesc appear consistently across vendors?**
   **PILOT-CONFIRMED**: free-text but reproducible per row. Match key uses
   it as-is. No standardization required.

3. **Are PO# and BL# required, recommended, or rare?**
   **PILOT-CONFIRMED**: NOT OBSERVED in pilot. Removed from match-key
   variants. invoiceNo + ourRefNo + vendorInternalId cover all cases.

4. **What's the expected batch volume per year?**
   **PILOT-CONFIRMED**: ~3,400 ERP rows/year. Trivial for IndexedDB.
   SharePoint sizing for Phase 3 is also non-issue at this volume.

### STILL OPEN — blocks spec freeze

5. **Are weekly statements re-issues or deltas?** (statement, not ERP)
   **PILOT-OPEN**: pilot did not include statement files. Statement
   re-issue vs delta determines:
   - STMT_LINE match key shape
   - Default dedup decision for statement rows
   - Whether statement comparison view (v2.3) needs cross-period
     reconciliation logic

   **Action**: user to provide 1-2 weekly statement files (LOCAL_USD,
   LOCAL_CAD, AGENT) before v2.4 spec freeze.

6. **Is May "CRDR" file a separate source type or a different report?** (PILOT-NEW)

   **Action**: user to confirm whether this is intentional ERP behavior

## Risks

| Risk | Mitigation |
|------|------------|
| AP costLineDesc missing in some files | Match-key falls back to AMBIGUOUS, surfaced to user |
| User uploads same batch twice (byte-identical) | sha256 check at upload — instant block with confirm-replace flow |
| User reverts a batch that other batches depended on | Cascade detection: warn + show affected batches |
| IndexedDB cleared by user (privacy mode etc.) | Mandatory weekly JSON snapshot export prompt |
| SharePoint adapter behaves differently from IndexedDB | Identical conformance test suite for both adapters |
| Aging reconciliation drift (asOfDate vs current state) | Snapshot the asOfDate result + checksum; verify on reload |
| **Filename-based source-type detection misclassifies "CRDR" file** (PILOT-NEW) | Schema-aware detection: column count + first-cell value enum |
| **Existing v2.0-2.3 baseline JSON has untagged CRDR rows** (PILOT-NEW) | Migration script in v2.4-15 re-tags as CRDR_MEMO |
| **AP costLineDesc free-text variation** (e.g. "TRUCKING COST" vs "TRUCKING CHARGE COST") | Match key uses exact string match. If real-world produces minor variations, surface as AMBIGUOUS for user decision |
| **Statement weekly behavior unknown** (PILOT-OPEN) | Pilot before code: cannot freeze STMT_LINE match key without statement data |
