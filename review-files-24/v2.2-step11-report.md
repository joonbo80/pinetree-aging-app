## What this step ships

### Per-tab CSV export

Click "Export CSV ↓" on any tab footer to download a UTF-8 BOM CSV
matching what's visible.

| Tab | CSV columns |
|-----|-------------|
| Transactions | partyKey/partyName + sourceType/direction/currency/signedBalance/rawBalance/isZeroBalance + agingDays/agingBucket/agingBasisDate + invoiceDate/postDate + department + ourRefNo/invoiceNo/crdrNo/vendorInvoiceNo/vendorName/blNo + flags + sourceFile/sheet/row + transactionId |
| Statements | partyKey/partyName + source(LOCAL/AGENT) + currency/statementBalance + matchType + matchTypeLabel(human-readable) + ourRefNo/invoiceNo/crdrNo + matchedTransactionId + matchedTxSourceFile/Row/Signed + delta + referenceStatus/differenceType + sourceFile/Row |
| Reviews | partyKey/partyName + category/reasonCode/severity + reason + currency/amount + transactionId + sourceFile/Sheet/Row + reviewItemId |
| Duplicates | partyKey/partyName + groupIdentityKey/Currency/Count/PotentialImpact + memberTransactionId + memberSourceType/Direction/SignedBalance/InvoiceDate/PostDate + memberRefNos + memberSourceFile/Sheet/Row |

Every CSV row carries partyKey + partyName (v2.1.1 invariant) so
multi-party CSV merges reconcile cleanly.

### Filename convention

```
aging-party-{partyKey}-{tab}-{YYYYMMDD-HHmm}.csv
```

Example: `aging-party-skymaster-express-transactions-20260505-1432.csv`

### Statements CSV - strict vs broad labels carry through

The Step 7 broad/strict NOT_IN_ERP distinction is preserved in CSV:

- Strict (review-item-backed) rows -> matchTypeLabel = "Confirmed not in ERP"
- Broad (outside-extract) rows -> matchTypeLabel = "Not in uploaded ERP extract"

Both keep matchType = NOT_IN_ERP_EXTRACT (raw enum) so downstream
analysis tools can also parse the structured value.

### Duplicates CSV - flattened

One row per (group, member). Group columns repeat across the group's
members so a pivot-by-groupIdentityKey reproduces the UI grouping.
For iata-cargo-accounts: 9 groups x 2 members = 18 body rows (verified).

### Empty tabs - header-only CSV

Empty tab still produces a valid CSV (BOM + header row, no body).
Verified on a1-intermodal (statement-only party with 0 transactions
and 0 duplicate groups).

## File split rationale

Per reviewer Step 10 note about file size, csvExport.ts is a separate
module from the start. Step 11's test set K explicitly verifies:

- csvExport.ts exists as separate file
- csvExport does NOT import from PartyDetailPage (no back-reference)
- csvExport imports types from parsing-engine (correct dependency direction)

This establishes the pattern for future splits: feature modules
(export, filtering, etc.) live alongside PartyDetailPage but stand
independently. Step 12-13 may continue this trend.

## Methodology note

Step 11 demonstrates the **file split as work progresses** pattern.
Rather than letting PartyDetailPage.tsx grow unbounded and doing a
big-bang split at v2.3, each new feature module (csvExport here) goes
into its own file from day one. Test set K pins the boundary so future
PRs cannot accidentally re-merge.

This is the same discipline as invariant-pinning user boundary fixes:
once we have the right architecture, we encode it as a test so it
can't drift.

Step 12 will be a one-line link addition in Dashboard. Step 13 is
browser hands-on. Both should be 1-round absorbs.
