## Findings -- answers to spec draft open questions

### Q1. Are weekly statements re-issues or deltas?

**ERP data answer: DELTA-only.**

Evidence (all from `sources_analyzed.*.internalId_overlap_count` in JSON):
- INVOICE internalId: baseline 16822-17359 -> May 17360-17377. Overlap: 0.
- AP internalId: baseline 29398-29944 -> May 29945-29989. Overlap: 0.
- INVOICE business ID (PEIN sequence): baseline ...19479 -> May 19480-19500. Overlap: 0 (`sources_analyzed.INVOICE.businessId_overlap_count`).
- Date ranges: baseline 2026-01-01 to 2026-04-30, May 2026-05-01 to 2026-05-05. No date overlap.

Every weekly cut contains ONLY new transactions issued since the last
upload. The ERP assigns sequential internalIds, so byte-identical
re-uploads are easily detected.

**Statement answer: still unknown.** Pilot did not include LOCAL/AGENT
statement files. Statement re-issue vs delta behavior is the remaining
open question.

### Q2. Are AP costLineCode patterns standardized?

No. The costLineDesc field is free-text (sample values from pilot rows):

Implication: match key uses costLineDesc + amount + currency. Free-text
is fine as long as it's reproducible across uploads of the same row.

### Q3. PO# / BL# / invoiceNo / ourRefNo frequency in real data?

From `field_presence_pilot` in JSON:

| Field | Source | Present (count / total / pct) |
|------|--------|-------------------------------|

## CRITICAL FINDING #1 -- AP multi-cost-line CONFIRMED

User's reviewer-flagged concern is fully validated. Numbers from
`ap_multi_cost_pattern` in JSON:

```
total_distinct_ourRefNos:     33
multi_cost_line_shipments:     7
single_cost_line_shipments:   26
multi_cost_ratio_pct:         21.2%
```

Sample (from `ap_multi_cost_pattern.samples_first_3`):
```
ourRefNo "WSATOR261111" -> 3 cost lines:
  WIN YAN LOGISTICS    -165.0 USD  DETENTION CHARGE COV   (vInv 29989)
  HAPAG LLOYD CANADA    165.0 USD  DETENTION CHARGE 1 D   (vInv 29967)
  WIN YAN LOGISTICS     900.0 CAD  TRUCKING COST          (vInv 29958)
```

**21.2% of pilot AP shipments have 2+ cost lines** sharing the same
ourRefNo. If v2.4 dedup uses `ourRefNo + amount + currency` alone, it
would silently drop the cost lines that happen to share an amount,
even though they're legitimately distinct (different vendor + different
cost type). This is the silent-corruption class.

Resolved: AP_LINE match key MUST include `vendorName + costLineDesc`
in addition to `amount + currency`. Documented in v2.4-spec-draft-rev2.

Note on rev1 number: rev1 prose said "17.5% (40 AP rows / 7 shipments)"
which was wrong -- 40 was raw AP row count, not the dedup-relevant
denominator. Correct denominator is 33 distinct ourRefNos. Reviewer
caught this.

## CRITICAL FINDING #2 -- "CRDR" file is NOT one source type

This was unexpected. The baseline `2_CRDR_JAN-APR_2026.xls` and the
pilot `2-1_CRDR_MAY_1-5_2026.xls` files share the same filename pattern
but contain completely different data:

```
BASELINE (Jan-Apr CRDR):
  - 19 columns, 346 rows
  - Business ID col 1 = "PECDR016797" (CR/DR memo numbers)
  - Vendor col 2 = "SINO LOGISTICS (MALAYSIA)"
  - Has invoice-style aging columns (post date, due date, amounts)
  - This is CR/DR memos issued -- accounting documents

PILOT (May 1-5 "CRDR"):
  - 15 columns, 7 rows
  - Business ID col 1 = "EFT" / "E-TRANSFER" (payment method!)
  - Party col 2 = "PANTOS LOGISTICS CANADA INC."
  - Has bank/cheque columns ("HANA KOREA EXCHANGE BANK", "CHEQUE ISSUED : RBC 5431")
  - This is PAYMENT ENTRIES -- transactions, not memos
```

These are two different domain entities sharing one filename pattern.

The baseline CRDR file = "credit/debit memos" (a kind of accounting
document). The May "CRDR" file = "cash receipts / disbursements"
(payment register entries).

Both might legitimately be called "CRDR" colloquially, but they have
different schemas, different keys, and different match rules.

### Implications for v2.4 spec

1. Parser cannot trust filename alone -- must inspect column structure
   to determine actual source type.
2. Source type taxonomy needs to expand:
   - INVOICE (existing)
   - CRDR_MEMO (was just CRDR in v2.0-2.3 -- the baseline kind)
   - PAYMENT_REGISTER (NEW -- the May kind, EFT/cheque records)
   - AP (existing)
3. v2.4 step v2.4-3 (match key construction) must add
   PAYMENT_REGISTER_LINE as a new match-key variant.
4. Backward compat: v2.0-2.3 baseline JSON has 346 "CRDR" rows that
   are actually CRDR_MEMO. A migration step must re-tag them.

### Why this matters for Phase 3

If v2.4 ships with single CRDR source type and SharePoint adapter is
built on that, Phase 3 will require schema migration in production.
Catching this NOW saves that pain.

### Question for user

Is the May "CRDR" file actually meant to be a separate source type, or
is it the SAME ledger but a different report view that the ERP exports?
Need to confirm before v2.4 spec freeze.

## Cross-source linking confirmed

`ourRefNo` is the natural shipment identifier across INVOICE / AP. In
the May pilot (numbers from `cross_source_linking_in_pilot` in JSON):

```
invoice_distinct_ourRefNos:  14
ap_distinct_ourRefNos:       33
shared_ourRefNos:            11
```

**11 ourRefNos appear in BOTH INVOICE and AP** -- cross-source links.

Sample shared refs (from `cross_source_linking_in_pilot.sample_shared_refs`):
```
919JSELG32447900   -> INVOICE 1045 CAD + AP WIN YAN 900 CAD
919JSELG39048600   -> INVOICE 1045 CAD + AP WIN YAN 900 CAD
HJTS023107         -> INVOICE 470 CAD + AP AES CORE 248.85 CAD
KR17501865         -> INVOICE 660 USD + AP SWIRE 660 USD
RSTOR2603021       -> INVOICE 870 CAD + AP CN-ACC 0 CAD + AP WIN YAN 725 CAD
```

This is the expected pattern: a shipment generates one INVOICE (revenue
side) and 1+ AP entries (cost side). v2.4 should preserve this linking
-- Party Detail already does it via the existing ourRefNo field, no
change needed for that aspect.

Note on rev1 numbers: rev1 prose said "12 / 27 / 8" which was wrong
on all three. Correct numbers from JSON: 14 invoice / 33 AP / 11
shared. Reviewer caught the divergence.

## Methodology note

This pilot accomplished what spec freezing on assumptions cannot: it
surfaced 2 critical findings (AP multi-line proof, CRDR-is-two-types)
that would have caused expensive rework in v2.4 implementation. The
CRDR finding alone is worth the entire pilot session -- without real
data, no amount of spec discussion would have caught it.

For Phase 3 SharePoint design, the same pattern should apply: pilot
with a small SharePoint List and 1 week of synthetic batches BEFORE
locking the schema, to surface unexpected domain rules.

The rev2 self-correction (prose-vs-JSON divergence) reinforces a second
methodology rule: in any artifact pair where a machine-readable file
and a human-readable narrative coexist, the machine-readable file is
the source of truth and prose must cite it explicitly. Re-derived
numbers in prose are a silent-corruption class.
