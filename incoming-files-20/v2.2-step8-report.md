### Spec compliance

| Spec rule | Implementation |
|-----------|---------------|
| No dedup (a tx can appear in multiple categories) | Filter logic does NOT dedupe by transactionId; baseline has 80 such tx and they all surface |
| Default filter: all categories | useState<...>('ALL') |
| Cross-tab focus on linked tx | erp-match-link button when item.transactionId resolves; em-dash when null (NOT_IN_ERP_EXTRACT) |
| No rawRow display | Trace panel never references rawRow |
| Per-currency totals (D5) | Two separate accumulators, never summed |
| Statement-only party | Tab still renders; review items often present even without ERP tx |

### Category priority sort

Most-actionable first, mirroring Review Queue card order:

```
1. NOT_IN_ERP_EXTRACT  - audit-grade missing rows
2. AGING_90_PLUS       - real money outstanding
3. DUPLICATES          - potential double-counting
4. WARNINGS            - parser-flagged anomalies
5. UNKNOWN_DEPARTMENT  - housekeeping
```

Within each category: |amount| desc, then id stable.

For win-yan-logistics (13 reviews across 3 categories):
- First sorted row: NOT_IN_ERP_EXTRACT (priority 1) — the 1 strict review
- Then: 6 DUPLICATES (priority 3) — sorted by |amount|
- Then: 6 WARNINGS (priority 4)

The accountant sees confirmed-missing first, then likely double-counts,
then flagged anomalies. Same scan-down ergonomics as the Review Queue
index page.

### Cross-tab focus

Linked transaction column ("Tx") shows:

- A "-> AP" / "-> INVOICE" / "-> CRDR" link button when item has
  transactionId AND that tx is in the party's set
- An em-dash when null (NOT_IN_ERP_EXTRACT items, by data fact)

Clicking the link calls onFocusTransaction(matchedTx.id), reusing the
Step 7 mechanism: switch to Transactions tab, OPEN-filter exception,
scrollIntoView, 1.8s flash highlight.

Trace panel also has an explicit "View this transaction in
Transactions tab ->" button for the same effect from inside the
expanded panel.

### Trace panel completeness

Fields shown:
- Source File / Source Row / Source Sheet (when present)
- Category, Reason Code, Severity
- Party Name (v2.1.1 invariant)
- Party Key (mono)
- Currency, Amount (when present)
- Reason text (full, in dedicated REASON section)
- LINKED TRANSACTION section (when matchedTx)
- DETAILS section (extra primitive fields from item.details)

rawRow is NEVER shown (spec §1.7).

## Methodology note

Step 8 went straight to code (no micro-spec) per user direction. In
hindsight this was correct — Reviews tab had:

- All UI patterns established by Step 6/7 (filter bar, table, trace,
  cross-tab focus)
- All business rules already specified (no dedup, all categories
  visible, per-currency totals, cross-tab focus when transactionId)
- All real-data shapes already understood

So no new design decisions required pre-coding. Steps 6 and 7 had
genuine micro-spec questions; Step 8 was assembly with reused parts.

This is a useful pattern: spec-freeze big design choices early
(Steps 1-3), code each tab with established patterns (Steps 6-9), and
budget review rounds where genuinely new mechanisms appear (Step 6
OPEN filter, Step 7 broad/strict labels and focus mechanics).

Step 9 (Duplicates) and Step 10 (trace unification) should be even
smaller — both are largely cleanup and consistency rounds on top of
the patterns now in place.
