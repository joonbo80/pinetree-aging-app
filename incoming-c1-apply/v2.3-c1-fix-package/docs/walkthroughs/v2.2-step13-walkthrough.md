## Purpose

Step 13 is a HUMAN walkthrough, not a code task. It exercises the full
v2.2 surface area as a user would, with the logo + favicon visible,
and surfaces any visual or interaction issue that unit tests cannot
catch.

This is the last gate before v2.2 release tag. Anything not caught
here will be caught by users in production.

## Pre-flight setup

Order matters. Run in this exact sequence -- if any step fails, stop
and fix before moving on. Note: dev tokens shown below are
local-development-only. See security note at end of this section.

1. Open PowerShell in the workspace directory.

2. Run logo invariant test FIRST (catches structural regressions before
   spending time on build):
   ```powershell
   node .\tools\test-logo-integration.mjs
   ```
   Expected: `PASS: 32    FAIL: 0`. If FAIL, stop and report -- do
   not proceed to build.

3. Confirm clean build:
   ```powershell
   cd aging-app
   npm.cmd run build
   ```
   Expected: clean build, no TypeScript errors, no missing-asset
   warnings.

4. Check port 5173 occupancy (LISTENING sockets only):
   ```powershell
   netstat -ano | findstr ":5173" | findstr LISTENING
   ```
   or preferably:
   ```powershell
   Get-NetTCPConnection -LocalPort 5173 -State Listen
   ```
   If a listener is found, identify the holder:
   ```powershell
   Get-Process -Id <PID>
   ```
   If it is a stale node from an earlier dev session, stop only
   that single PID:
   ```powershell
   Stop-Process -Id <PID> -Force
   ```
   Avoid the bulk command `Get-Process node | Stop-Process -Force`
   because it may kill unrelated Node tools (VS Code, Claude Code,
   etc).

5. In a second PowerShell window, start the API server with the
   required environment variables set in the SAME PowerShell session
   (env vars only persist within the session that set them):
   ```powershell
   cd "C:\Users\samue\OneDrive\Documents\New project\aging-api"
   $env:AGING_UPLOAD_TOKEN="dev-secret-change-me"
   $env:AGING_PYTHON="C:\Users\samue\.cache\codex-runtimes\codex-primary-runtime\dependencies\python\python.exe"
   $env:AGING_PHASE1_ROOT="C:\Users\samue\OneDrive\Documents\New project"
   npm.cmd run dev
   ```
   Expected output should include:
   ```
   [aging-api] listening on http://127.0.0.1:3001
   CORS allowlist: ... http://localhost:5173 ...
   ```
   Verify port 5173 is in the CORS allowlist and that the output
   does NOT contain `upload auth: NOT CONFIGURED`. If that string
   appears, AGING_UPLOAD_TOKEN was not picked up; re-set in the same
   PowerShell session and restart the API server.

6. Back in the first PowerShell window, start the UI dev server
   with the matching token env var set in the SAME session:
   ```powershell
   cd "C:\Users\samue\OneDrive\Documents\New project\aging-app"
   $env:VITE_AGING_UPLOAD_TOKEN="dev-secret-change-me"
   npm.cmd run dev
   ```
   Expected: Vite output shows `Local: http://localhost:5173/`
   exactly. If it falls back to 5174, return to step 4 (a stale
   process still holds 5173).

7. Open browser at the dev server URL: http://localhost:5173

8. Open browser DevTools (F12). Keep Console tab visible during
   walkthrough. Any RED error there is a finding.

### Scenario 4 -- Party Detail page deep dive (15 min)

For at least 5 parties picked in Scenario 3, walk each party detail
page through:

#### 4a. Header card (Step 4)

- [ ] Party name + key shown
- [ ] Six summary cards: total amount, currencies, transaction count,
      statements count, review items, oldest unpaid date
- [ ] Card numbers reconcile against Dashboard Top Parties row

#### 4b. Tabs (Step 5)

- [ ] Tabs: Transactions / Statements / Reviews / Duplicates (4 tabs)
- [ ] Tab navigation works -- clicking a tab loads its content
- [ ] Tab keyboard nav: arrow keys, Enter to activate (a11y)

#### 4c. Transactions tab (Step 6)

- [ ] All transactions for the party visible
- [ ] Sort works on date, amount, source columns
- [ ] Filter or search box (if present) narrows the list correctly
- [ ] Each transaction row clickable -> trace expansion (if Step 6
      includes per-row trace, verify it expands inline without
      breaking layout)
- [ ] Match-type badges visible and color-coded correctly

#### 4d. Statements tab (Step 7)

- [ ] Statements for the party listed
- [ ] Match indicator (matched / unmatched / partial) visible
- [ ] Click a match cell -> jumps to corresponding transaction (if
      cross-tab navigation is in scope)

#### 4e. CSV export (Step 11)

- [ ] CSV export button visible on Party Detail
- [ ] Click triggers a CSV download
- [ ] Downloaded file opens correctly (Excel / Numbers / text editor)
- [ ] CSV rows match the visible transaction list
- [ ] CSV does NOT include columns that should be hidden

The Step 11 download stability fix should make the second download in
the same session also work. So:

- [ ] Click CSV export TWICE in a row -- both downloads should succeed
      (not just the first)

CSV encoding verification (accounting-user-critical):

- [ ] Pick a party whose name OR transaction memo contains Korean
      characters or special symbols (search baseline data for at
      least one such party)
- [ ] Open the downloaded CSV directly in Excel (not via text editor first)
- [ ] Korean characters and special symbols (comma in name, quotes,
      etc) display correctly in Excel cells -- not as garbled text
      (typical mojibake symptoms: a row of question marks, hex-escape
      sequences, or random CJK-looking characters that do not match
      the expected name)
- [ ] Row count in Excel matches the visible transaction count on
      Party Detail
- [ ] If the file uses CSV semicolons or commas, fields with commas
      are properly quoted (not split across columns)

Why this matters: csvExport.ts writes a UTF-8 BOM (\uFEFF) so Excel
recognizes the encoding. If the BOM is stripped (by mail attachment,
by SharePoint upload, by file-system tools), Korean text breaks. This
test confirms the BOM survives the download path. If Excel garbles
Korean here, the BOM was lost somewhere between blob creation and
file save -- BLOCKER finding.

#### 4f. Visual polish

- [ ] No overflow scrollbars where there shouldn't be
- [ ] Logo and header chrome remain visible while scrolling Party
      Detail (sticky header working)
- [ ] Mobile-narrow viewport: try resizing browser to ~700px wide and
      confirm header logo doesn't overlap nav controls

## Methodology note

This Step 13 guide is designed to surface the silent-corruption class
of bugs that unit tests cannot catch: visual regressions, interaction
breakage, performance smells, real-browser asset loading.

The 32-invariant test suite from the logo work catches structural
issues. It cannot catch "the logo looks too small" or "the CSV button
is unreachable on mobile". Those need a human in front of a real
browser. Step 13 is exactly that.

For Phase 3 SharePoint work and beyond, similar hands-on gates should
exist before each release tag. Unit tests + invariants + hands-on =
three layers of defense.
