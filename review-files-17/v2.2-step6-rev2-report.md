## Fix 1 (P1 — Open balance filter)

This was the most important call from reviewer. My round-1 reasoning
("spec compliant, one filter click away") was wrong for an accounting
tool — first screen must show actionable balances.

### What changed

```ts
type TxDirectionFilter = 'ALL' | 'OPEN' | 'receivable' | 'payable' | 'settled';

const [direction, setDirection] = useState<TxDirectionFilter>('OPEN');  // default

if (direction === 'OPEN') {
  r = r.filter(t => t.direction === 'receivable' || t.direction === 'payable');
} else if (direction !== 'ALL') {
  r = r.filter(t => t.direction === direction);
}
```

Dropdown order also reordered so `Open` is first (most-used path):

```
Open (receivable + payable)   ← default
All
Receivable
Payable
Settled
```

### Verified on real data

SKYMASTER EXPRESS:
- All transactions: 86
- Settled: 34
- **OPEN (recv + pay): 52** ← what user sees by default
- Non-settled: 52 (matches OPEN)

The previous round's "settled rows dominate top-10 default sort" issue
**disappears automatically** when default = OPEN, because settled rows
are filtered out before sorting.

## Fix 2 (P1 — Date sort uses postDate fallback)

Single source of truth helper:

```ts
function displayDate(tx: PreviewTransaction): string {
  return tx.invoiceDate ?? tx.postDate ?? '';
}
```

Used by:
- Display cell: `{displayDate(tx) || '—'}`
- Sort modes: `displayDate(b).localeCompare(displayDate(a))` (desc)
- Sort modes: `displayDate(a).localeCompare(displayDate(b))` (asc)

### Verified on real data

CRDR sortable rows on SKYMASTER:
- With `invoiceDate` only: N rows
- With `invoiceDate ?? postDate` fallback: N+ rows (more rows now sort
  by their visible date)

The fallback recovers CRDR/AP rows that have `postDate` but no
`invoiceDate` — they no longer sort as empty strings.

## Fix 4 (P2 — Real button for expand control)

Replaced `<tr tabIndex={0} aria-expanded ...>` with `<tr>` containing
a proper `<button>` in the first cell:

```tsx
<tr className="tx-row">
  <td className="tx-col-toggle">
    <button
      type="button"
      className="tx-toggle-btn"
      onClick={...}
      onKeyDown={escapeHandler}
      aria-expanded={isExpanded}
      aria-controls={`tx-trace-${tx.id}`}
      aria-label={`${isExpanded ? 'Collapse' : 'Expand'} trace for ${tx.sourceType} row ${tx.trace.sourceRow}`}
    >
      <span className="tx-toggle-chevron" aria-hidden="true">{isExpanded ? '▾' : '▸'}</span>
      {tx.sourceType}
    </button>
  </td>
  ...
</tr>
{isExpanded && (
  <tr className="tx-trace-row" id={`tx-trace-${tx.id}`}>
    ...
  </tr>
)}
```

Why this is better:
- Screen readers announce "expand button" with the row context
- Native Enter/Space activation (no manual `onKeyDown`)
- Esc on the button still collapses (kept that handler)
- `aria-controls` connects the button to its trace panel `<tr id>`
- `aria-label` includes both action verb and identifying context
- Visual chevron `▸/▾` provides sighted users the affordance

The row itself is no longer focusable; focus is properly anchored on
the toggle button.

## Reviewer's structural insight applied

Reviewer noted the spec D14 freeze argument was insufficient because
"the app's purpose is AR/AP verification". The fix preserves spec
compliance (Direction filter still has all 5 options including ALL)
while making the **default** match the app's purpose.

This is a UX default, not a spec change. The `OPEN` virtual option
being on the same dropdown as the literal direction values is a clean
way to expose the most-used view without forking the filter model.

---

## Bonus: my round-1 mistake worth noting

I argued in round 1 that "settled rows dominating top-10 is spec-
compliant, document and move on". Reviewer correctly pushed back: in
an audit-grade tool, **defaults matter as much as capabilities**.

Lesson for future rounds: when a spec choice produces a UX result that
contradicts the app's stated purpose, escalate to a spec discussion
before declaring it acceptable. "Spec frozen" is not a license to
ship counter-purpose defaults.
