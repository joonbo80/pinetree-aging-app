## 4. Known Issues (Not Release Blockers)

### 4.1 [P1] LEMOND alias normalization (deferred to v2.3/v2.4)

Symptom: LEMOND Party Detail shows 142 transactions but 0 statements,
even though 67 statement rows exist for the same business in the
baseline data.

Root cause: partyKey for the same business differs between data
sources.

- transactions[].partyKey = "lemond-food-corp-new-addr"
- statementLinks[].partyKey = "lemond-food"

Party Detail uses exact partyKey matching in selectPartyDetail, which
is the intended behavior per current spec. The split is a data-level
alias issue, not a UI or wire-up bug.

Other large parties (WIN YAN, SKYMASTER, IATA) do not exhibit this
because their partyKey is consistent across transactions and statement
links.

Decision: filed as P1 in v2.3/v2.4 backlog (Section 5). Not a v2.2
release blocker because:

- UI mechanics work as specified.
- Code does what the spec says.
- A hard-coded merge of "lemond-food" + "lemond-food-corp-new-addr"
  would be unsafe without a systematic alias normalization design,
  which belongs in v2.4 spec rev6 (party-normalization area).

### 4.2 [POLISH] Dashboard placeholder message clarity

Symptom: When user navigates directly to /dashboard before clicking
Confirm Import on a preview, the page shows "Dashboard placeholder.
Load or upload data in Upload + Parsing Preview first." This is
technically correct (preview data is not committed) but ambiguous
to the user.

Suggested fix: change copy to "Your last import is in preview state
and was not committed. Either go back to confirm it, or load fresh
data." Filed in v2.3 backlog.

### 4.3 [POLISH] API offline produces 4 console errors

Symptom: When the API server is not running, the app attempts to
fetch /api/health and /api/parse-upload with retry, producing four
RED CORS errors in DevTools.

Impact: cosmetic. The header shows "API OFFLINE" correctly. The
UI does not break.

Suggested fix: when API is reachable but failing, downgrade retry
errors from console.error to console.warn. Filed in v2.3 backlog.

### 4.4 [POLISH] Vite build chunk size warning

Symptom: vite-reporter warns "Some chunks are larger than 500 kB
after minification."

Impact: build succeeds. First-load size could be reduced.

Suggested fix: configure rollupOptions.output.manualChunks to
split PartyDetailPage and the parser into separate chunks. Filed
in v2.3 backlog.

### 4.5 [GUIDE] Step 13 walkthrough rev3 has small gaps

Discovered during execution:

- Preflight did not instruct user to check port 5173 occupancy
  (Vite falls back to 5174 if 5173 is held by another process,
  which breaks API CORS allowlist).
- Preflight did not instruct user to start the API server (aging-api
  npm run dev) as a separate process.
- Scenario 2 did not call out the explicit Confirm Import button
  click after LOAD BASELINE DEMO; without it, /dashboard direct
  URL shows placeholder rather than data.

Action: file as walkthrough rev4 update. Does not affect v2.2
release.
