## What was wrong with rev2

The CSS file was created and the test verified it existed and contained
the right classes. But nothing imported it. The build produced a CSS
bundle that did NOT contain `.tx-toggle-btn`, `.party-tab-filter-bar`,
or `.pill-receivable` - the runtime UI was unstyled.

The test passed because it only asserted file existence. The reviewer
correctly called this out as a vacuous-pass: same shape as the v2.1.1
silent partyName collapse pattern (file present, behavior wrong).

Lesson: when adding new infrastructure (CSS file, config, build step),
the test must assert the LOAD chain, not just the file's presence.

## Fix 1: global.css imports v2.2-party-detail.css

```css
/* aging-app/src/styles/global.css (top of file) */

/* v2.2 Party Detail styles (Step 4-6).
   Loaded ahead of the rest so any global utility classes can override
   v2.2-specific defaults if needed. */
@import './v2.2-party-detail.css';
```

One line added. Vite/Rollup will bundle the imported CSS into the same
output stylesheet, so `.tx-toggle-btn` and friends will be in
`dist/assets/index-XXXX.css` after build.

## Fix 2: Test now asserts the LOAD chain, not just file existence

Two new assertion blocks in `tools/test-party-detail-page-step6.mjs`:

### L.5.a - Source-level @import check

```js
const importsViaAtRule =
  /@import\s+['"]\.\/v2\.2-party-detail\.css['"]/.test(globalCss) ||
  /@import\s+url\(['"]?\.\/v2\.2-party-detail\.css['"]?\)/.test(globalCss);
const inlinedKey =
  globalCss.includes('.tx-toggle-btn') &&
  globalCss.includes('.party-tab-filter-bar');
check(
  `global.css loads v2.2 CSS (via @import OR inlined classes)`,
  importsViaAtRule || inlinedKey,
);
```

This catches the case where someone (human or AI) creates a CSS file
without wiring it into the app. Either an @import or the classes being
inlined directly into global.css will pass; nothing else will.

### L.5.b - Built-bundle check (when dist exists)

```js
const distAssetsPath = resolve(__dirname, '../aging-app/dist/assets');
if (statSync(distAssetsPath).isDirectory()) {
  const cssFiles = readdirSync(distAssetsPath).filter(f => f.endsWith('.css'));
  if (cssFiles.length > 0) {
    const bundled = cssFiles
      .map(f => readFileSync(resolve(distAssetsPath, f), 'utf-8'))
      .join('\n');
    check(`built CSS bundle contains .tx-toggle-btn`,        bundled.includes('.tx-toggle-btn'));
    check(`built CSS bundle contains .party-tab-filter-bar`, bundled.includes('.party-tab-filter-bar'));
    check(`built CSS bundle contains .pill-receivable`,      bundled.includes('.pill-receivable'));
    check(`built CSS bundle contains .aging-90plus`,         bundled.includes('.aging-90plus'));
  }
}
```

When a dist build exists, this enforces that the imported CSS actually
made it through the bundler. When dist is missing (e.g. test runs
before first build), the source-level check above is the binding
assertion and a console info note is printed.

This is the exact failure mode the reviewer caught: file present,
import missing, dist bundle missing the classes. The check now
catches each link in that chain.

## Reviewer's structural insight applied (again)

Same pattern as v2.1.1: the test that catches the issue must be
binding, not advisory. File existence is advisory. @import is binding.
Bundled CSS is the strongest binding (it asserts the entire build
pipeline did its job).

The L.5 block is now the strongest of any v2.2 invariant set, which is
appropriate since CSS load chain failures are silent: the build still
passes, the test still passes, the user just sees an unstyled UI.
