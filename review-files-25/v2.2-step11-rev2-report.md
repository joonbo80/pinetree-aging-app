## Reviewer P2 finding closed

Title: Object URL is revoked immediately after click

Symptom: Some browsers (notably Firefox) intermittently cancel the
download because the blob URL is freed before the download pipeline
consumes it. This is silent failure — the accountant clicks Export
CSV, nothing happens, the data is lost without an error.

Fix applied (csvExport.ts):

```ts
function triggerDownload(csv: string, filename: string): void {
  const blob = new Blob([csv], { type: 'text/csv;charset=utf-8;' });
  const url = URL.createObjectURL(blob);
  const a = document.createElement('a');
  a.href = url;
  a.download = filename;
  a.style.display = 'none';
  document.body.appendChild(a);   // (1) DOM-attach for Firefox click() reliability
  a.click();
  a.remove();                     // (2) clean up DOM
  window.setTimeout(() => URL.revokeObjectURL(url), 0);  // (3) defer revoke
}
```

Three changes that work together:

1. **DOM-attach the anchor** — Firefox's click() can no-op on
   detached anchors in some versions.
2. **Remove the anchor** after clicking so the DOM stays clean.
3. **Defer URL.revokeObjectURL via setTimeout** so the current event
   loop completes the click + browser download dispatch before the
   blob URL is freed.

## New invariants (set L) — 5 added

The reviewer specifically asked for "a small test that looks for
deferred revoke to keep this stable". I added 5 invariants in set L
that pin each part of the fix:

```
L. Browser download stability (P2 fix)
  ✅ anchor is appended to document.body before click
  ✅ anchor is removed after click
  ✅ URL.revokeObjectURL is deferred via setTimeout
  ✅ revoke is NOT called synchronously after click (the bug pattern)
  ✅ anchor uses display:none to avoid layout flicker
```

The fourth check is the strongest — it's a NEGATIVE invariant that
fails if the synchronous-revoke pattern (`a.click(); URL.revokeObjectURL`)
ever returns. Future regressions cannot vacuous-pass.

This is the same audit-grade pattern that v2.1.1 partyName fix and
Step 6 CSS load chain used: when a silent-failure mode is identified,
the test must enforce the FIX, not just the presence of code.

## Methodology note

This is the second time a P2 reviewer finding led to invariant
strengthening rather than just code change. (First was Step 6 rev3
CSS load chain.) Pattern:

```
1. Reviewer identifies silent-failure mode
2. Code fix applied
3. Invariant test pins the FIX SHAPE (not just code presence)
4. Negative invariant added: "the bug pattern must NOT return"
```

This pattern is the audit-grade equivalent of a regression suite —
it doesn't just ensure the fix is in, it ensures the bug pattern
cannot silently return through future refactoring.

For Phase 3 design (Collection Action System), this same discipline
will matter: silent failure modes in email send / payment promise
tracking will need the same pinned-fix invariant pattern, because
those are even higher-stakes than CSV download.

## Bonus observation: not all silent-failure modes are equal

The CSV download bug is a relatively benign silent failure — accountant
notices "nothing happened" and tries again. By contrast:

- v2.1.1 partyName silent collapse: silent corruption, accountant
  uses wrong data without knowing
- Step 6 CSS not loaded: visible (unstyled UI) so users notice
- Step 11 deferred revoke: intermittent (works most of the time)

The intermittent class is worst-of-three because:
- Tests pass (no consistent reproduction)
- Users blame themselves ("I must have clicked wrong")
- Real reproduction needs cross-browser testing

The negative-pattern invariant is the right defense because it makes
the bug pattern itself the failure signal, not the bug occurring.
