## North Star

The app shifts from "single snapshot analyzer" to "time-series
accumulator." Weekly/monthly/quarterly/year-end uploads append to a
persistent store, deduplicated by reference number with explicit user
decisions, with full audit history. This is the foundation Phase 3
SharePoint persistence will sit on.

## Core concepts

### Import Batch

Every Excel upload creates ONE Import Batch. The batch is the unit of
de-duplication, audit, and rollback.

```typescript
interface ImportBatch {
  batchId: string;              // UUID
  uploadedAt: string;           // ISO 8601
  uploadedByUserId?: string;    // Phase 3 only -- null in v2.4
  parserVersion: string;        // e.g. "1.3.0"
  schemaVersion: string;        // e.g. "1.0"

  period: PeriodKey;
  cycle: StatementCycle;

  sourceFiles: Array<{
    fileName: string;
    fileSize: number;
    sourceType: 'INVOICE' | 'CRDR' | 'AP' | 'AGENT_STATEMENT' | 'LOCAL_STATEMENT_USD' | 'LOCAL_STATEMENT_CAD';
    rowCount: number;
    sha256: string;
  }>;

  dedupDecisions: DedupDecision[];

  status: 'draft' | 'committed' | 'reverted';
  committedAt?: string;
  revertedAt?: string;
  revertReason?: string;
}
```

### Period Key (ISO 8601 compatible)

```typescript
type PeriodKey =
  | `${number}-W${number}`        // weekly:    "2026-W18"
  | `${number}-${number}`         // monthly:   "2026-04"
  | `${number}-Q${1|2|3|4}`       // quarterly: "2026-Q2"
  | `${number}-12-FY`             // yearend:   "2026-12-FY"
  | `${number}-${number}-${number}-AH`;  // adhoc:  "2026-05-15-AH"

type StatementCycle = 'weekly' | 'monthly' | 'quarterly' | 'yearend' | 'adhoc';
```

interface StatementPeriod {

### Match Key -- source-specific (CRITICAL, PILOT-CONFIRMED)

Per reviewer note and pilot data: AP has multi-line cost structure
that requires special handling.

PILOT-CONFIRMED: 21.2% of pilot AP shipments have 2+ cost lines
sharing the same `ourRefNo` (7 of 33 distinct ourRefNos in May pilot).
Sample:
```
ourRefNo "WSATOR261111":
  WIN YAN     -165 USD  DETENTION CHARGE COV
  HAPAG LLOYD  165 USD  DETENTION CHARGE 1 D
  WIN YAN      900 CAD  TRUCKING COST
```

PILOT-CONFIRMED: BL# and PO# fields are NOT present in pilot data.
Removed from match key variants.

```typescript
type MatchKey =
  | { type: 'INVOICE_LINE';
      invoiceNo: string;
      amount: number;
      currency: 'USD'|'CAD';
      party: string; }

  | { type: 'CRDR_LINE';
      crdrNo: string;               // commonly PECDR..., but legacy IDs (e.g. DAS...) accepted
      amount: number;
      currency: 'USD'|'CAD';
      party: string; }

  | { type: 'AP_LINE';
      vendorInternalId: number;
      ourRefNo: string;
      vendorName: string;
      costLineDesc: string;
      amount: number;
      currency: 'USD'|'CAD'; }

  | { type: 'STMT_LINE';            // PILOT-OPEN
      statementRef: string;
      amount: number;
      currency: 'USD'|'CAD'; }

  | { type: 'AMBIGUOUS';
      reason: string;
      fallbackFields: Record<string, string>; };
```

CRITICAL DOMAIN RULE: AP_LINE matching uses vendorName + costLineDesc.
Two rows with the same ourRefNo + amount but different vendor or cost
line are DISTINCT and must NOT be auto-deduplicated.

### Source Type Taxonomy

The 6 source types from rev1 are correct. Schema-aware parser
detection is retained as a defensive measure for future unknown
formats:

| File pattern | Source type | Schema | Detection rule |
|--------------|------------|--------|----------------|
| `*INVOICE*.xls` | INVOICE | 26 cols, business ID = "PEIN..." | filename + col count |
| `*CRDR*.xls` | CRDR | 19 cols, CRDR memo shape (date / party / DR / CR / balance / currency / internalId) | filename + col count + shape match (see CRDR detection note below) |
| `*AP*.xls` | AP | 25 cols, col[1] = vendor name | filename + col count |
| `*AGENT*STATEMENT*.xls` | AGENT_STATEMENT | varies | filename pattern |
| `*LOCAL*STATEMENT*USD*.xls` | LOCAL_STATEMENT_USD | varies | filename pattern |
| `*LOCAL*STATEMENT*CAD*.xls` | LOCAL_STATEMENT_CAD | varies | filename pattern |

If a file's schema does not match the expected source type, the parser
surfaces the mismatch as an AMBIGUOUS error rather than silently
misinterpreting the data. This defensive layer is what would have
caught the rev2 wrongly-uploaded file at parse time -- the parser
would have refused to interpret a 15-column file as a 19-column CRDR
source.

#### CRDR detection note (rev5)

CRDR business IDs are commonly in the `PECDR...` format, but legacy
or non-standard IDs exist. In the baseline (Jan-Apr 2026, 346 rows):

- 345 rows = `PECDR...` format (99.7%)
- 1 row   = `DAS2602042` format (0.3%, valid CRDR row, same 19-column shape)

Detection rules MUST NOT reject CRDR rows solely because the business
ID prefix is not `PECDR`. The rule shape is:

```
CRDR row is valid IF:
  - file pattern matches *CRDR*.xls AND
  - column count = 19 AND
  - shape match: col[0]=date, col[1]=businessId(non-empty),
                 col[2]=party, col[7]=amount(numeric),
                 col[16]=currency in {USD,CAD},
                 col[18]=internalId(integer)

CRDR row is NOT rejected because:
  - businessId starts with something other than PECDR (legacy IDs OK)
  - businessId is shorter or longer than typical PECDR length
```

If a CRDR row fails the shape match (e.g. col[7] is non-numeric, or
col[18] is missing), the parser surfaces it as AMBIGUOUS and asks the
user. It does NOT silently drop the row.

This rule was added in rev5 after reviewer caught the `DAS2602042`
case. The lesson: prefix patterns are useful as DESCRIPTIVE
documentation but unsafe as ACCEPTANCE criteria. Use shape match as
the gate; use prefix patterns as hints only.

## Methodology

### Pilot-before-freeze (still standing)

For any new domain entity, pilot 1 real batch before spec freeze. The
ERP pilot caught the AP multi-line silent-corruption case.

### Verify the file before introducing a new source type (rev4 NEW)

If a file's schema is unexpectedly different from the baseline of the
same source type, verify with the user whether the uploaded file is
the intended source BEFORE introducing a new source type, match key
variant, or migration step.

This was the rev2/rev3 mistake: an unexpected schema in a file labeled
"CRDR" was interpreted as evidence of a second source type
(PAYMENT_REGISTER), and a full set of spec changes was made to
accommodate it. The actual cause was the user accidentally uploading
a different file.

The verification step: when a finding seems to suggest a new source
type, ask the user "is this the file you meant to upload, or did you
intend to upload a different file?" before proceeding to change the
spec.

This belongs to the same family as v2.1.1 partyName silent corruption,
Step 6 CSS load chain, Step 11 download stability, and rev2 prose vs
JSON divergence: documentation makes a claim about reality; verify
the claim against reality, not against intent. The rev2 spec made a
claim about real source data; that claim should have been verified
with the user before being baked in.

### Reduce silent-corruption windows (rev4 NEW)

When a hypothesis is later shown to be wrong, roll back the spec
immediately rather than waiting to bundle the rollback with other
changes. The cost of leaving a wrong hypothesis in a draft spec is
that downstream readers may treat it as confirmed. This is the
rationale for issuing rev4 immediately on rollback rather than
waiting for statement data.

### Prefix patterns describe, shape patterns gate (rev5 NEW)

When data has a "common ID format" (e.g. `PECDR...` for CRDR rows),
that format is useful as DESCRIPTIVE documentation but unsafe as
ACCEPTANCE criteria. Real-world data contains legacy or non-standard
IDs at low frequency (the baseline CRDR has 1 `DAS...` ID among 346
rows = 0.3%).

Detection/validation rules MUST gate on SHAPE (column count, field
types, structural consistency), not on PREFIX patterns. Prefix
patterns belong in user-facing docs as hints, not in parser accept/
reject logic.

This is the same family of silent-corruption rules as v2.1.1
partyName, Step 6 CSS load chain, Step 11 download stability, and
the rev2/rev3 prose-vs-JSON divergence: a rule that "usually works"
on the visible data may silently reject the rare valid case if
applied as a hard gate.
