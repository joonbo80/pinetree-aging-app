## RETRACTED FINDING -- "CRDR is two source types"

The rev2/rev3 finding that "CRDR" filename pattern represented two
distinct source types (CRDR_MEMO vs PAYMENT_REGISTER) is RETRACTED.

What happened:
1. User uploaded a file labeled `2-1_CRDR_MAY_1-5_2026.xls`.
2. The uploaded file had 15 columns and contained EFT/E-TRANSFER/CHEQUE
   payment register entries.
3. I (Claude) interpreted the schema mismatch as evidence of a second
   source type and updated the spec rev2 with PAYMENT_REGISTER as a
   new source type, PAYMENT_REGISTER_LINE match key, and a
   v2.0-2.3-baseline migration step.
4. Reviewer P1/P2 fixes for rev3 made the spec internally consistent
   with this hypothesis.
5. User identified the upload mistake and re-uploaded the actual May
   CRDR file.
6. Re-uploaded file: 19 columns, PECDR ID sequence continuous from
   baseline, identical schema. Same source type as baseline CRDR.

Lesson: when a finding seems to suggest a new source type, verify with
the user that the uploaded file is the intended source BEFORE
proceeding to spec changes. This is now documented in the v2.4 spec
methodology section.

## Methodology notes

### Pilot accomplished what spec assumptions cannot

The pilot caught the AP multi-line silent-corruption case via real
data. This finding survives rev4 because it was independently
confirmed in the AP file, not the disputed CRDR file.

### What the rev2->rev4 rollback teaches

The pilot also produced a wrong finding (CRDR-is-two-types) that
required rev4 rollback. The wrong finding was caused by trusting an
uploaded file as authoritative without verifying with the user that
the file was the intended one.

The new rule: when a pilot file's schema is unexpectedly different,
verify with the user before introducing a new source type. This is
now in the v2.4 spec methodology section.

### Source of truth discipline

The rev2 prose-vs-JSON divergence (caught by reviewer P1 #2) and the
rev4 rollback are both examples of the same pattern: documentation
makes a claim about reality; that claim must be verified against
reality. The verification step is what catches silent-corruption-class
bugs at design time instead of in production.

### Reducing silent-corruption windows

rev4 was issued immediately on rollback rather than waiting for
statement data, because leaving a wrong hypothesis in a draft spec
risks downstream readers treating it as confirmed.
