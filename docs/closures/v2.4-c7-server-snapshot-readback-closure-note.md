## 1. Status

C7 Server Snapshot Read-back is closed.

C7 completed the first end-to-end Phase 3 persistence loop:

```text
Browser-local C3 snapshot
  -> aging-api upload endpoint
  -> SharePoint document library
  -> aging-api latest snapshot endpoint
  -> browser localStorage restore
  -> /aging hard-load Workbench restore
```

## 2. Purpose

C6 proved that the application can upload a C3 snapshot envelope to
SharePoint through `aging-api`.

C7 proves the reverse path: the application can retrieve the latest
server-side snapshot from SharePoint, validate the C3 envelope, save it
back to browser storage, and restore the Statement Collection Workbench
on hard-load.

This is the first practical proof of single-user multi-device
persistence.

## 3. Implementation Summary

### 3.1 API

Endpoint:

```text
GET /api/snapshot/latest
Authorization: Bearer <Microsoft Graph delegated token>
```

Implementation:

- Lists files in SharePoint folder `AGING&STATEMENT APP`.
- Filters to `aging-snapshot-*.json`.
- Sorts by SharePoint `lastModifiedDateTime` descending.
- Downloads the latest file.
- Parses JSON.
- Validates the C3 snapshot envelope.
- Returns the snapshot payload to the browser.

### 3.2 Frontend

The `/spike/auth` route now includes:

```text
Restore latest server snapshot
```

The button:

- Requires a signed-in Microsoft delegated token.
- Calls `GET /api/snapshot/latest`.
- Parses the returned snapshot through existing C3 snapshot logic.
- Saves the snapshot to `localStorage`.
- Displays restore summary fields.

## 5. Snapshot Selection Rule

C7 intentionally uses a simple latest-file rule:

1. List SharePoint folder children.
2. Keep files whose names start with `aging-snapshot-`.
3. Keep files whose names end with `.json`.
4. Sort by `lastModifiedDateTime` descending.
5. Download the first file.

This is sufficient for the first single-user multi-device proof.

Future work can add a snapshot picker, immutable history display,
manual delete, or workspace diff.

## 6. Standing Rules Preserved

C7 preserves:

- C3 snapshot envelope shape.
- Browser-local fallback.
- Microsoft Entra delegated token path from C5.
- SharePoint upload target from C6.
- No workflow metadata writes.
- No Owner/Memo/Promise implementation.
- No multi-user edit or conflict resolution.

## 7. Known Boundaries

C7 does not solve:

- Snapshot version picker.
- Snapshot conflict resolution.
- Multi-user editing.
- SharePoint list metadata integration.
- Owner assignment.
- Memo persistence.
- Payment promise tracking.

Those remain future v2.4+ tasks.

- C5 delegated auth is working.
