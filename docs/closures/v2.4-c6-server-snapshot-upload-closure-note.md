## 1. Status

C6 Server Snapshot Upload is closed.

C6 proved the first real Phase 3 server persistence operation:

```text
C3 browser-local snapshot envelope
  -> aging-api
  -> Microsoft Graph
  -> SharePoint AGING&STATEMENT APP folder
```

Browser-local C3 persistence remains the fallback. C6 adds server
upload; it does not yet add server read-back.

## 2. Scope Delivered

C6 delivered:

- A server snapshot upload endpoint.
- Snapshot envelope validation.
- SharePoint upload using the C5 delegated Graph auth path.
- A dev/spike UI control for uploading the current browser-local C3
  snapshot.
- Documentation of the successful upload proof.

C6 did not deliver:

- Snapshot restore from SharePoint.
- Snapshot listing.
- Workflow metadata list writes.
- Owner, memo, or promise tracking.
- Multi-user edit.
- Conflict resolution.

## 4. API Contract

Endpoint:

```text
POST /api/snapshot/upload
Authorization: Bearer <Microsoft Graph delegated token>
Content-Type: application/json
```

Request body:

```json
{
  "filename": "aging-snapshot-2026-05-01-batch-id.json",
  "snapshot": { "snapshotVersion": "v2.3-c3-snapshot-v1" }
}
```

Response body:

```json
{
  "status": "ok",
  "source": "sharepoint-snapshot-upload",
  "filename": "...",
  "folder": "AGING&STATEMENT APP",
  "file": {
    "id": "...",
    "name": "...",
    "size": 1798433,
    "webUrl": "..."
  }
}
```

## 5. Validation

The API validates:

- Bearer token through the C5 delegated Graph auth boundary.
- Snapshot object exists.
- Snapshot version is `v2.3-c3-snapshot-v1`.
- `summary.importBatchId` exists.
- `summary.asOfDate` exists.
- `result.uploadSession` exists.
- Filename ends with `.json`.
- Filename has no unsafe path segments.

## 9. Lessons

### 9.1 Server Upload Needs a Local Snapshot First

The first upload attempt correctly failed because no browser-local C3
snapshot existed yet.

Defense: C6 UI reports a clear message:

```text
No browser-local C3 snapshot found. Load baseline or files, then Confirm Import first.
```

### 9.2 Upload Without Read-back Is Not a Full Sync Loop

C6 proves server persistence write. It does not yet prove restore from
server.

Defense: C7 should be Server Snapshot Read-back before workflow metadata
or owner/memo features.

## 10. Next Step

Recommended next sub-task:

```text
C7 Server Snapshot Read-back
```

C7 should:

- List or locate the uploaded snapshot in SharePoint.
- Download the JSON through `aging-api`.
- Parse the C3 snapshot envelope.
- Restore it into the app.
- Keep browser-local C3 fallback.

After C7, the project will have a complete single-user multi-device
persistence loop:

```text
Upload snapshot to SharePoint
Download snapshot from SharePoint
Restore workspace on another device/browser
```
