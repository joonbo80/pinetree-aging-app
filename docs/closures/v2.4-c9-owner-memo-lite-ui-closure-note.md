## 1. Status

C9 Owner/Memo Lite UI is closed.

C9 added the first production-facing workflow metadata UI on top of the
SharePoint list surface proven in C8.

## 2. Purpose

C8 proved that `aging-api` can create, read, update, and delete
workflow metadata rows in SharePoint.

C9 connects that metadata surface to the Statement Collection Workbench
so a collector can save lightweight follow-up context:

```text
Owner
Memo
```

Promise tracking remains deferred.

## 3. User Flow

Verified flow:

1. Sign in on `/spike/auth`.
2. Open `/aging`.
3. Expand a party rollup row.
4. Enter Owner/Memo.
5. Save.
6. Receive `Saved to SharePoint.`
7. Reopen/reload and see the saved values remain.

## 4. Implementation Summary

### 4.1 API

New production-facing workflow endpoints:

```text
GET  /api/workflow/item?workflowKey=...
POST /api/workflow/item
```

Both endpoints require a Microsoft Graph delegated token.

The API:

- Resolves the SharePoint workflow list.
- Maps SharePoint display names to internal field names.
- Reads workflow metadata by `workflowKey`.
- Creates a row if missing.
- Updates the row if present.

### 4.2 UI

The Owner/Memo panel appears inside expanded party rollup rows in the
Statement Collection Workbench.

This keeps the first workflow UI close to the party row that collectors
are already reviewing.

## 5. Workflow Key

Workflow metadata rows are keyed by:

```text
{workspaceId}__{partyKey}__{currency}__{direction}
```

This preserves:

- Party separation.
- Currency separation.
- AR/AP direction separation.

## 6. Verified Result

Browser smoke result:

```text
Saved to SharePoint.
```

The saved Owner/Memo values remained visible after reload/reopen.

## 7. Important Finding

Initial C9 save failed with:

```text
Sign in on /spike/auth first.
```

Root cause:

The C5 spike token was stored only in `sessionStorage`. Browser
`sessionStorage` is tab-scoped, so signing in on `/spike/auth` in one
tab did not make the token available to `/aging` in another tab.

Fix:

The C5 spike token is now also stored in `localStorage`. The C9 UI reads
`localStorage` first, then falls back to `sessionStorage`.

This is acceptable for the local Phase 3 spike path. A production auth
implementation should replace this with a real MSAL/Entra session
strategy.

## 8. Standing Rules Preserved

C9 preserves:

- C5 delegated auth path.
- C8 SharePoint list field-name mapping.
- Browser-local snapshot fallback.
- Party/currency/direction separation.
- No Promise tracking UI yet.
- No multi-user conflict resolution.

## 9. Known Boundaries

C9 does not solve:

- Promise date/amount/status editing.
- Owner picker from tenant users.
- Memo history.
- Conflict detection.
- Bulk assignment.
- Production-grade MSAL session management.

Those remain future tasks.
