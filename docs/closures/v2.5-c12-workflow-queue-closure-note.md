## 1. Status

C12 Workflow Queue is closed.

C12 adds the first v2.5 daily collector view:

```text
Committed C3/C7 snapshot party rollup
  + SharePoint workflow metadata from v2.4
  -> read-only /workflow queue
  -> filters, summary counts, and party navigation
```

The queue is intentionally read-only. C13 remains the boundary for
inline owner, memo, promise, contacted, snooze, or bulk workflow
actions.

## 2. Product Purpose

v2.4 made workflow metadata persistent:

- C8 stores metadata in the `PE Aging&Statement Workflow` SharePoint
  list.
- C9 stores owner and memo.
- C10 stores promise date, amount, and status.
- C11 generates email drafts.

C12 turns that stored metadata into a daily collector surface. The
collector can now open one screen and see which party/currency/direction
rows have owner, memo, promise, or follow-up context without editing the
underlying workflow data from that screen.

The product purpose is a calm queue, not an alert dashboard.

## 3. Implementation Summary

### 3.1 Route

New route:

```text
/workflow
```

The route is registered in `aging-app/src/App.tsx` and opens
`WorkflowQueuePage`.

The Dashboard now includes an entry point:

```text
Open Workflow Queue
```

### 3.2 Page

New page:

```text
aging-app/src/pages/WorkflowQueuePage.tsx
```

The page:

- Requires an existing committed snapshot.
- Builds the party rollup with `selectAgingReport`.
- Reads SharePoint workflow metadata through `apiClient.readWorkflowItems`.
- Uses the C5 delegated auth token from the existing auth spike storage.
- Merges snapshot rollup rows with workflow rows.
- Shows a read-only table.
- Links each row back to Party Detail through `Open Party`.

### 3.3 Merge Key

C12 preserves the accounting separation rules by merging on:

```text
workspaceId + partyKey + currency + direction
```

This prevents silent mixing of:

- USD and CAD.
- AR and AP.
- Same party name but different accounting direction.

### 3.4 Server-first With Cache Fallback

Workflow metadata is fetched from SharePoint first.

If the fetch fails, the page may fall back to the last cached workflow
metadata snapshot under:

```text
agingApp.workflowQueueCache.v1.{workspaceId}
```

When cached data is used, the page must show a visible stale indicator.
No stale workflow merge should be silent.

## 4. Filters

C12 includes filter chips for:

- Owner: `Me`, `Unassigned`, `All`.
- Due: `All`, `Overdue`, `Today`, `This week`, `Has promise`,
  `No promise`.
- Currency: `All`, `USD`, `CAD`.
- Direction: `All`, `AR`, `AP`, `Settled`.
- Memo: `Any`, `Has memo`, `No memo`.
- Status: `Active`, `All`.

The final first-open defaults are:

```text
Owner: All
Due: All
Currency: All
Direction: All
Memo: Any
Status: Active
```

## 5. Important Design Decision

The initial spec draft used `Owner: Me` and a narrower due filter as the
default. Browser review showed that this could hide useful rows on first
open because current owner values are still free text.

J changed the default to `Owner: All` and `Due: All`.

This is now the locked C12 behavior:

- First open should show the full queue.
- The user chooses when to narrow the queue.
- `Me` remains available as an explicit filter.
- Future owner picker or `ownerUserId` support can revisit the default.

This follows the v2.3 design philosophy:

```text
Calm accounting workbench, not alert dashboard.
```

For C12 that means:

```text
Do not pre-decide which workflow rows the collector should see.
```

## 8. Standing Rules Maintained

C12 maintains the standing rules established in v2.3 and v2.4:

- USD and CAD remain separated.
- AR and AP are never netted.
- `NOT_IN_ERP_EXTRACT` remains outside the Workflow Queue.
- Browser-local snapshot fallback remains valid.
- SharePoint workflow metadata is additive, not a replacement for the
  committed snapshot.
- The queue is read-only until C13.
- Calm visual tone remains the default.

## 9. Known Boundaries

C12 does not include:

- Inline owner change.
- Inline memo edit.
- Inline promise update.
- Mark contacted.
- Snooze.
- Bulk operations.
- Notifications.
- Email sending.
- New SharePoint columns.
- Owner identity picker or `ownerUserId` enforcement.

These are future sub-tasks, with C13 being the natural next owner for
inline actions.

## 10. Residual Risks

### 10.1 Owner Identity

`Owner: Me` currently depends on the stored owner value matching the
signed-in user representation. Because the current workflow UI allows
free-text owner values, `Owner: Me` can return zero rows even when a
human-readable owner is present.

Mitigation:

- Default to `Owner: All`.
- Keep `Me` as explicit filter.
- Defer owner picker / `ownerUserId` normalization to a future sub-task.

### 10.2 SharePoint Cache Staleness

If SharePoint fetch fails and local cache is used, stale workflow data
can be shown.

Mitigation:

- Visible stale indicator.
- No silent stale merge.

### 10.3 Orphan Workflow Rows

SharePoint workflow rows may exist for parties not present in the
current snapshot.

Mitigation:

- C12 does not display orphan workflow rows as normal queue rows.
- Orphan surfacing can be expanded later if it becomes operationally
important.

C12 is closed because it meets the intended v2.5 first milestone:

- v2.4 workflow metadata is visible in a daily queue.
- The queue merges with snapshot rollups.
- Accounting separation rules are preserved.
- The view is read-only.
- The first-open filter behavior is transparent.
- Automated checks and browser smoke passed.
