# v2.5 Release Candidate Note

Date: 2026-05-16

Tag candidate: v2.5-rc1
Expected final tag: v2.5.0
Primary sub-task: C12 Workflow Queue
Parent release: v2.4.0

## 1. Release Status

v2.5 is the first workflow presentation release for the Pinetree
AGING APP.

v2.4 made workflow metadata durable in SharePoint. v2.5 C12 turns that
metadata into a daily collector queue:

```text
SharePoint workflow metadata
  + committed snapshot party rollup
  -> read-only /workflow queue
  -> filters, summary counts, and party navigation
```

The release candidate is intentionally narrow. It does not add new
workflow writes. It proves that the stored owner, memo, promise, and
email-context layer can be viewed as a calm daily work queue.

## 2. Product Definition

v2.5 does not automate collections.

It adds a read-only work surface that helps the collector answer:

1. Which party rows have workflow context?
2. Which promises are due or upcoming?
3. Which rows are owned, unassigned, or assigned to someone else?
4. Which rows should I open next in the Workbench or Party Detail?

The screen follows the existing product identity:

```text
Calm accounting workbench, not alert dashboard.
```

- Inline memo edit.

- Mark contacted.

- Notifications.
- Email sending.
- New SharePoint columns.

Those are candidates for C13 or later.

## 4. Architecture

### 4.1 Route

```

Registered in:

```

```text

```

### 4.2 Data Sources

C12 merges two existing data streams.

Committed snapshot party rollup:

```text
selectAgingReport()
```

SharePoint workflow metadata:

```text
apiClient.readWorkflowItems(workspaceId, token)
```

Auth token source:

```text
loadC5AuthSpikeAccessToken()
```

### 4.3 Merge Key

The workflow queue uses the same accounting-safe key pattern established
in v2.4:

```text
workspaceId + partyKey + currency + direction
```

This prevents accidental merge of:

- USD and CAD.
- AR and AP.
- same party name across different accounting directions.

### 4.4 Cache Fallback

SharePoint workflow rows are fetched server-first.

If SharePoint fetch fails, the page can use:

```text
agingApp.workflowQueueCache.v1.{workspaceId}
```

The page must show a stale indicator when cached workflow metadata is
used. Silent stale data is not allowed.

## 5. Important Product Decision

The C12 spec originally considered a more opinionated first-open view
that defaulted toward the current user's rows and nearer due dates.

Browser review changed the default to a transparent first-open view:

```text
Owner: All
Due: All
Currency: All
Direction: All
Memo: Any
Status: Active
```

Reason:

- Owner values are currently free text.
- `Owner: Me` can return zero rows when the saved owner is a short name
  such as `J` instead of the Microsoft display name.
- First open should not hide valid workflow rows.
- The collector should decide when to filter.

`Owner: Me` remains available as an explicit filter. A future owner
picker or `ownerUserId` normalization can revisit the default.

## 8. Standing Rules Maintained

v2.5 C12 preserves the standing rules established across v2.3 and
v2.4:

- USD and CAD remain separated.
- AR and AP are never netted.
- `NOT_IN_ERP_EXTRACT` remains outside the Workflow Queue.
- Browser-local snapshot fallback remains valid.
- SharePoint workflow metadata is additive, not a replacement for the
  committed snapshot.
- C12 is read-only.
- Calm visual tone remains the default.

## 9. Known Boundaries

### 9.1 Owner Identity

`Owner: Me` depends on matching the signed-in Microsoft user to the
stored owner field. Since owner values are currently free text, the
filter may not match short names or initials.

Current mitigation:

```text
Default Owner filter = All
```

Future improvement:

```text
ownerUserId / owner picker
```

### 9.2 Inline Actions

C12 deliberately avoids write actions from the queue. This keeps v2.5.0
low risk and preserves a single write path for owner, memo, promise, and
email context.

C13 can add inline actions after C12 proves queue value.

### 9.3 SharePoint Cache Staleness

If the SharePoint workflow fetch fails and cached workflow rows are
used, the page must show the stale state.
