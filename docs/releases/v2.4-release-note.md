# v2.4 Release Note

Date: 2026-05-16
Status: FINAL
Tag: v2.4.0
Parent roadmap: `docs/specs/v2.4-phase3-persistence-roadmap.md`

## 1. Release Status

v2.4.0 is the first Phase 3 persistence release for the Pinetree
AGING APP.

The release moves the application beyond browser-local workflow state
and proves the practical SharePoint-backed path:

```text
Microsoft Entra delegated sign-in
  -> aging-api token validation
  -> Microsoft Graph / SharePoint access
  -> server snapshot upload
  -> server snapshot read-back
  -> SharePoint workflow metadata list
  -> owner, memo, promise, and email draft workflow
```

v2.4.0 keeps the v2.3 product identity:

```text
Calm accounting workbench, not alert dashboard.
```

## 2. Product Definition

v2.4.0 does not turn the AGING APP into an automatic collection system.

It adds the first durable workflow layer so the collector can preserve
context across browser sessions and devices while still reviewing and
deciding actions manually.

The release answers these practical workflow questions:

- Can the app sign in with Microsoft Entra?
- Can the API validate the delegated Microsoft token?
- Can the API write and read snapshots through SharePoint?
- Can a collector store owner, memo, and promise context per party row?
- Can collapsed rows show whether workflow context exists?
- Can the app prepare a safe email draft without sending automatically?

## 4. Scope Summary

### 4.1 C5 Entra Auth Spike

C5 proved Microsoft Entra delegated sign-in for the Pinetree tenant.

Verified results:

- Browser sign-in works.
- Signed-in user is displayed in `/spike/auth`.
- `aging-api` validates the delegated token.
- `GET /api/spike/whoami` returns the signed-in user context.
- SharePoint Graph calls can be made once site permissions are aligned.

Important implementation lesson:

The token audience for Microsoft Graph is:

```text
00000003-0000-0000-c000-000000000000
```

The API validation path must accept the Graph delegated token shape used
by this spike.

### 4.2 C6 Server Snapshot Upload

C6 proved that a browser-local C3 snapshot can be uploaded through
`aging-api` into the SharePoint document folder.

Verified result:

```text
Server snapshot upload
Status: ok
Source: sharepoint-snapshot-upload
Folder: AGING&STATEMENT APP
Size: 1798433
```

This made SharePoint the first server-side storage surface for the
AGING APP.

### 4.3 C7 Server Snapshot Read-back

C7 proved the reverse path:

```text
SharePoint snapshot
  -> aging-api latest snapshot endpoint
  -> browser localStorage
  -> /aging Workbench restore
```

Verified result:

```text

Status: ok
Source: sharepoint-snapshot-latest
Transactions: 1230
As of date: 2026-05-01
Local storage saved: yes
```

This closes the first single-user multi-device persistence loop.

### 4.4 C8 Workflow Metadata List Integration

C8 proved SharePoint list read/write for workflow metadata.

Verified list:

```text
PE Aging&Statement Workflow
```

Verified operations:

- Create item.
- Read item.
- Update memo/status fields.
- Delete smoke-test item.

Important implementation lesson:

SharePoint list display names are not always the same as internal field
names. The API must discover field internal names and write by internal
name.

### 4.5 C9 Owner/Memo Lite

C9 added editable workflow context inside expanded aging party rows:

```text
Owner
Memo
```

The data is saved to SharePoint through `aging-api` and reloaded from
the workflow metadata list.

This is intentionally lite:

- No owner picker yet.
- No memo history yet.
- No multi-user conflict handling yet.

### 4.6 C9.1 Row Indicators

C9.1 made workflow metadata visible from collapsed rows.

Collapsed row indicators:

```text
Owner: ...
Memo
```

This prevents hidden workflow data from being invisible in the main
collector table.

### 4.7 C10 Promise Tracking Lite

C10 added promise tracking fields:

```text
Promise Date
Promise Amount
Promise Status
```

Collapsed rows can show a Promise badge.

The v2.4.0 final polish normalizes SharePoint date values for display:

```text
2026-05-15T07:00:00Z -> 2026-05-15
```

The raw SharePoint value can still include time. The user-facing display
uses the date-only value.

### 4.8 C11 Email Draft Lite

C11 generates a collection email draft from:

- Party name.
- Currency.
- Direction.
- Open amount.
- Invoice count.
- Oldest aging.
- Statement status.
- Promise context.
- Internal memo context.

Actions:

```text
Copy Draft
Open Email Draft
```

C11 does not send email automatically.

`Open Email Draft` uses the local machine's `mailto:` handler. If
Windows or Chrome routes `mailto:` to a browser instead of Outlook, this
is a workstation configuration issue, not an application data blocker.
`Copy Draft` remains the safe fallback.

## 5. Accounting Safety Rules Preserved

v2.4.0 preserves the accounting rules established in v2.3:

- USD and CAD are never combined.
- AR/AP direction is never netted.
- Workflow metadata keys include party, currency, and direction.
- NOT_IN_ERP_EXTRACT remains outside the Aging Workbench party-rollup
  workflow.
- Email drafts require user review.
- No automatic send action exists.
- Browser-local C3 snapshot remains as fallback.

## 9. Known Boundaries

v2.4.0 does not include:

- Automatic email sending.
- Microsoft Graph `Mail.Send`.
- Recipient/contact lookup.
- Owner picker from tenant users.
- Promise history.
- Multi-user conflict resolution.
- Workflow metadata filtering.
- Snapshot picker/history UI.
- K1 alias normalization implementation.

These are candidates for v2.5+.

## 10. Release Decision

v2.4.0 is accepted as FINAL.

The release is production-useful because it proves the Phase 3
SharePoint persistence path while preserving manual control for
collection work.
