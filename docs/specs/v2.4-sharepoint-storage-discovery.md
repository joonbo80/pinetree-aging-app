## 1. Status

This discovery document is the required environment check before Phase 3 implementation begins.

The v2.4 Phase 3 roadmap chooses a SharePoint + `aging-api` middle-tier architecture. Before writing the C5 Entra Auth Spike or C6 Server Snapshot Upload micro-spec, the project needs real facts about the Pinetree Microsoft 365 / SharePoint environment.

Roadmap status remains DRAFT until this discovery is answered.

## 2. Discovery Goal

Confirm whether Pinetree's SharePoint environment can support:

- storing C3 snapshot JSON files in a document library,
- storing owner/memo/promise metadata in a SharePoint list,
- authenticating users through Microsoft Entra ID,
- allowing `aging-api` to read/write SharePoint data safely,
- preserving browser-local C3 snapshot as fallback.

## 3. Current Architecture Assumption

The planned Phase 3 shape is:

```text
aging-app
  |
  | authenticated calls
  v
aging-api
  |
  | Microsoft Graph / SharePoint API
  v
SharePoint
  |-- Document Library: snapshot JSON envelopes
  |-- List: owner/memo/promise metadata
```

This discovery either confirms that shape or identifies constraints that require changing it.

### 5.2 Delegated vs Application Permissions

Two possible models:

1. Delegated permissions:
   - API acts on behalf of the signed-in user.
   - SharePoint permissions follow the user's access.
   - Better audit alignment.

2. Application permissions:
   - API acts as an application/service principal.
   - Requires stronger admin control.
   - Easier background jobs, but broader access risk.

Questions:

- Which model is acceptable to Pinetree IT?
- Is per-user SharePoint audit required?
- Can a service principal be granted access only to the target site/library/list?
