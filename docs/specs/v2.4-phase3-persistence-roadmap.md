# v2.4 Phase 3 Persistence Roadmap

Date: 2026-05-13
Status: DRAFT
Parent release: v2.3.1
Primary direction: SharePoint/server-backed persistence foundation

## 1. Status

This document defines the v2.4 direction for Phase 3 persistence.

The goal is to extend the browser-local C3 Snapshot model into a server-backed persistence foundation that can later support owner assignment, memo history, promise tracking, read-only sharing, and eventually controlled multi-user collaboration.

This is a roadmap-level micro-spec. It does not implement Phase 3 directly. Each implementation phase should receive its own smaller micro-spec before code changes begin.

## 2. Direction

v2.4 should prioritize persistence foundation before collector workflow features.

### 2.1 Goal

Extend browser-local C3 Snapshot persistence into SharePoint/server-backed persistence.

The app should keep the C3 browser-local snapshot behavior while adding a path to sync or upload the same workspace state through the API middle tier.

### 2.2 Why Foundation First

Owner assignment, memo history, and promise tracking all require durable storage. Implementing those features only in browser-local state would create a second migration later when server persistence becomes available.

The safer sequence is:

1. Keep C3 local snapshot as the continuity layer.
2. Add server upload/download around the C3 snapshot envelope.
3. Add structured workflow metadata after the persistence foundation is clear.

### 2.3 Not Yet

The following are out of scope for v2.4 foundation planning:

- Full multi-user editing.
- Real-time collaboration.
- Fuzzy alias normalization.
- Payment workflow automation.
- Automatic party merge decisions.
- Replacing the C3 local snapshot path.

K1 remains on WATCHLIST until May statement files provide stronger evidence.

## 3. Decisions

| Question | Decision | Rationale |
|---|---|---|
| Server architecture | SharePoint + middle-tier (`aging-api` extension) | Keeps SharePoint details out of the frontend and reuses the current API boundary. |
| Authentication | Microsoft Entra ID | Natural fit for Microsoft 365 / SharePoint environment and replaces dev-token patterns. |
| Storage model | Both SharePoint document library and SharePoint list | Snapshot JSON belongs in documents; workflow metadata belongs in structured list rows. |
| Migration path | C3 snapshot to server upload endpoint | Progressive migration with low disruption. |
| Collaboration scope | Single-user multi-device first, read-only sharing later | Avoids edit conflicts while proving the persistence path. |

## 4. Architecture Overview

Phase 3 should use a middle-tier architecture:

```text
aging-app
  |
  | HTTPS + Entra-authenticated session
  v
aging-api
  |
  | Microsoft Graph / SharePoint API
  v
SharePoint
  |-- Document Library: snapshot JSON envelopes
  |-- List: owner/memo/promise metadata
```

### 4.1 Frontend Role

The frontend should:

- Continue to load and save C3 local snapshots.
- Call `aging-api` for server persistence operations.
- Display server sync status.
- Avoid direct SharePoint REST or Graph calls in v2.4.

### 4.2 API Role

The API should:

- Validate authenticated user identity.
- Accept snapshot upload requests.
- Store snapshot JSON in SharePoint document library.
- Retrieve the latest snapshot for the authenticated user.
- Later expose structured metadata endpoints for owner/memo/promise data.

### 4.3 SharePoint Role

SharePoint should be the durable storage system, not the application logic layer.

It should store:

- Snapshot JSON documents.
- Structured workflow metadata rows.
- Basic audit fields such as created by, modified by, and timestamps.

## 5. Authentication

Phase 3 should replace local dev-token assumptions with Microsoft Entra ID.

### 5.1 Target Model

The target auth model is:

1. User signs in through Microsoft Entra ID.
2. The app obtains or receives an authenticated session.
3. `aging-api` validates the user identity.
4. `aging-api` accesses SharePoint using the approved application model.

### 5.2 Dev Mode

The existing dev-token pattern can remain only for local development.

Production Phase 3 behavior should not depend on:

- `dev-secret-change-me`
- unauthenticated upload routes
- manually shared tokens

### 5.3 Open Auth Questions

These require a later auth-specific micro-spec:

- Whether the frontend uses MSAL directly.
- Whether `aging-api` validates bearer tokens directly.
- Whether the API uses delegated permissions or application permissions to SharePoint.
- How local development auth is simulated.

## 6. Storage Design

Phase 3 storage should separate bulk snapshot state from structured workflow metadata.

### 6.1 Snapshot Document Library

Use a SharePoint document library for full snapshot JSON envelopes.

Snapshot documents should store the C3 envelope shape:

```text
snapshotVersion
appVersion
exportedAt
dataSource
summary
result
```

The current C3 version is:

```text
v2.3-c3-snapshot-v1
```

### 6.2 Snapshot Document Metadata

Each snapshot document should include enough metadata to query and display it:

- Workspace ID.
- User ID or owner principal.
- Display name.
- asOfDate.
- importBatchId.
- transaction count.
- statement link count.
- createdAt.
- createdBy.
- snapshotVersion.
- appVersion.

### 6.3 Workflow Metadata List

Use a SharePoint list for structured workflow metadata.

Potential list columns:

- workspaceId
- partyKey
- partyName
- ownerUserId
- ownerDisplayName
- memoText
- promiseDate
- promiseAmount
- promiseCurrency
- promiseStatus
- updatedAt
- updatedBy

Owner/memo/promise fields should not be embedded only inside the snapshot JSON if they need to be queried, filtered, or updated independently.

### 6.4 Why Both

Snapshot JSON is the right shape for restoring a workspace.

Structured SharePoint list rows are the right shape for:

- owner assignment,
- memo visibility,
- promise tracking,
- filtering by status,
- audit review,
- future dashboards.

## 7. Migration Path

The migration should be progressive and low-risk.

### 7.1 Phase 3-A: Server Upload Endpoint

Add an `aging-api` endpoint that accepts the C3 snapshot envelope and stores it server-side.

Initial behavior:

- Browser-local snapshot still works.
- User can upload/sync current snapshot to server.
- API validates envelope version and summary counts.
- API returns server snapshot ID and timestamp.

### 7.2 Phase 3-B: Server Read-Back

Add a way to retrieve the latest server snapshot for the authenticated user.

Initial behavior:

- User can restore from local snapshot.
- User can restore from server snapshot.
- UI clearly labels the source:
  - Local Snapshot
  - Server Snapshot
  - Uploaded Baseline

### 7.3 Phase 3-C: Dual Persistence

After upload/read-back is stable, allow optional dual persistence:

- save locally,
- sync to server,
- show sync timestamp,
- preserve offline tolerance when server is unavailable.

### 7.4 Phase 3-D: Workflow Metadata

Only after server snapshot persistence is stable, add structured metadata:

- owner assignment,
- memo,
- promise date/amount/status.

### 7.5 Phase 3-E: Sharing

Read-only sharing can follow after server persistence exists.

Multi-user editing remains out of scope until conflict rules are explicitly designed.

## 8. Collaboration Scope

v2.4 should target single-user multi-device sync first.

### 8.1 v2.4 Scope

The same authenticated user should be able to:

- upload a snapshot from one device,
- open the app on another device,
- restore the latest server snapshot,
- continue using browser-local fallback if server restore is unavailable.

### 8.2 Later Scope

Later read-only sharing can allow:

- user A uploads a snapshot,
- user B can view the snapshot,
- user B cannot edit workflow metadata unless granted permission.

### 8.3 Far Future

Full multi-user editing requires:

- conflict rules,
- record-level ownership,
- optimistic concurrency,
- audit trail,
- permission model,
- undo/rollback strategy.

It should not be smuggled into v2.4.

## 9. Risks

### 9.1 SharePoint API Quotas

Snapshot JSON may be large. Upload and download behavior should be measured before promising frequent sync.

### 9.2 Token Refresh

Entra token refresh and session expiration can break long-running collector workflows if not handled clearly.

### 9.3 Offline Mode

Collectors may need the app to keep working when server sync fails. C3 local snapshot should remain the fallback path.

### 9.4 Snapshot Versioning

Server snapshots must preserve `snapshotVersion` and `appVersion`. Future envelope versions require backward compatibility rules.

### 9.5 Conflict Scenarios

Even single-user multi-device sync can create conflicts if two devices modify metadata independently. v2.4 should avoid metadata write conflicts until rules exist.

### 9.6 Data Sensitivity

Snapshots contain accounting data. Server persistence requires explicit access control and storage governance.

## 10. Standing Rules

Phase 3 must preserve existing accounting rules:

- USD and CAD must not be merged.
- AR and AP must not be netted.
- C2 Workbench remains a calm accounting workbench, not an alert dashboard.
- NOT_IN_ERP_EXTRACT remains a Review Queue workflow, not a party rollup status.
- K1 alias normalization remains evidence-driven and manual until stronger data exists.
- Browser-local snapshot remains a valid fallback until server persistence is proven.

These names are placeholders for future micro-specs.

Output:

- Server restore path.
- UI source state.

### C8: Workflow Metadata Foundation

Goal:

- Define SharePoint list schema for owner/memo/promise metadata.
- Avoid implementing broad workflow UI until storage is proven.

Output:

- Metadata schema.
- Minimal create/read/update proof.

### C9: Owner/Memo Lite

Goal:

- Add owner and memo to party-level workflow after storage foundation exists.

Output:

- Party-level owner assignment.
- Party-level memo.
- Server-backed persistence.

## 12. Non-Goals

v2.4 should not include:

- fuzzy alias clustering,
- automatic party merges,
- full payment automation,
- Teams notification automation,
- mobile-first redesign,
- broad dashboard redesign,
- multi-user concurrent editing,
- replacing SharePoint with an unrelated database.

## 13. Success Criteria

Phase 3 roadmap is ready to proceed when:

- v2.4 direction is approved.
- C5-C9 candidate sub-tasks are accepted or adjusted.
- Auth model is chosen for the first implementation spike.
- SharePoint storage target is identified.
- C3 snapshot envelope compatibility is preserved.
- Existing v2.3/C3 invariants remain green.
