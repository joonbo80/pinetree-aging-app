# v2.4 C9 Owner/Memo Lite UI Micro-spec

Date: 2026-05-15
Status: CLOSED
Parent roadmap: `docs/specs/v2.4-phase3-persistence-roadmap.md`
Previous step: `docs/closures/v2.4-c8-workflow-metadata-list-integration-closure-note.md`

## 1. Goal

C9 adds the first production-facing workflow metadata UI.

Scope is intentionally limited to:

```text
Owner
Memo
```

Promise tracking remains out of scope for this first UI pass.

Closure evidence, 2026-05-15:

- Owner/Memo panel appears in expanded party rollup rows.
- Owner/Memo save returned `Saved to SharePoint.`
- Saved values remained visible after reload/reopen.
- Token sharing issue between `/spike/auth` and `/aging` tabs was fixed by storing the C5 spike token in localStorage as well as sessionStorage.

## 2. User Flow

1. J signs in on `/spike/auth`.
2. J restores or loads a snapshot.
3. J opens `/aging`.
4. J expands a party rollup row.
5. J edits Owner/Memo.
6. J saves to SharePoint.
7. Another browser/session can read the same metadata after sign-in.

## 3. Workflow Key

Workflow metadata rows are keyed by:

```text
{workspaceId}__{partyKey}__{currency}__{direction}
```

This preserves:

- Party separation.
- Currency separation.
- AR/AP direction separation.

## 4. API

New endpoints:

```text
GET  /api/workflow/item?workflowKey=...
POST /api/workflow/item
```

Both require Microsoft Graph delegated token.

## 5. UI Placement

C9 places the Owner/Memo Lite panel inside the expanded party rollup row.

This avoids adding a new global workflow surface before the metadata
model has real usage feedback.

## 6. Non-goals

C9 does not include:

- Promise date/amount/status UI.
- Conflict resolution.
- Multi-user editing.
- Audit history display.
- Bulk owner assignment.

## 7. Completion Criteria

C9 is complete when:

- API build passes.
- App build passes.
- Expanded party row loads metadata from SharePoint.
- Saving owner/memo creates or updates a workflow list item.
- Reloading the row reads the saved owner/memo back.
