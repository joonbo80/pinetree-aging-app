## 3. Spike Goal

Prove that an authenticated user can perform one round-trip
SharePoint operation through `aging-api` from the deployed app boundary.

The smallest acceptable proof:

1. User signs in through Microsoft Entra ID.
2. `aging-app` calls `aging-api` with the resulting identity.
3. `aging-api` validates the identity.
4. `aging-api` performs one SharePoint operation, either:
   - upload a tiny test JSON file to the snapshot folder, OR
   - read one row from the workflow list, OR
   - write one test row to the workflow list and delete it.
5. The result is returned to the user with the signed-in display name.

That round-trip is enough to retire the Outcome B auth unknowns.

## 4. Decision Questions

The following decisions belong inside C5 and must be answered before
C6 begins. Each decision should be recorded back into the roadmap.

### 4.1 Q1 App Registration Owner

| Option | Detail |
|---|---|
| A | J creates the app registration as M365 admin |
| B | Pinetree IT/admin owns the registration |

Recommendation: A. J is already the site owner and the AGING APP scope
is internal. A single owner reduces handoff risk during the spike.

### 4.2 Q2 Permission Model

| Option | Detail |
|---|---|
| A | Delegated permissions (API acts as the signed-in user) |
| B | Application permissions (API acts as a service principal) |
| C | Hybrid (delegated for v2.4, application later for background jobs) |

Recommendation: A for the spike. Delegated permissions align with
collector audit expectations. Application permissions are powerful but
require admin governance that v2.4 does not yet need.

### 4.3 Q3 Frontend Identity Flow

| Option | Detail |
|---|---|
| A | Frontend uses MSAL to sign in and obtains a token |
| B | Frontend logs in through `aging-api` which issues a session cookie |
| C | Frontend signs in via MSAL and `aging-api` validates the bearer token |

Recommendation: C. MSAL on the frontend obtains an access token. The
frontend sends it as a Bearer header. `aging-api` validates the token
and only then talks to SharePoint. This avoids inventing a custom
session model and keeps Microsoft 365 as the identity source of truth.

### 4.4 Q4 API SharePoint Access

| Option | Detail |
|---|---|
| A | API uses Microsoft Graph with the user's delegated token |
| B | API uses SharePoint REST API with the user's delegated token |
| C | API uses an application token (service principal) for SharePoint |

Recommendation: A. Microsoft Graph is the modern, supported entry point
for SharePoint document libraries and lists. Delegated Graph keeps audit
alignment with Q2.

### 4.5 Q5 Dev Mode Strategy

| Option | Detail |
|---|---|
| A | Keep `dev-secret-change-me` for local development only |
| B | Use a separate dev app registration with test users |
| C | Run the full Entra flow even locally |

Recommendation: B for after C5. Until C5 lands, keep A so local
development stays unblocked. C5 closure should pick B as the standing
dev mode.

### 4.6 Q6 Spike Output Scope

| Option | Detail |
|---|---|
| A | Auth proof only (identity round-trip) |
| B | Auth + one SharePoint read |
| C | Auth + one SharePoint write + cleanup |

Recommendation: C. A real SharePoint write retires the API write
unknown from Section 2.2. Cleanup avoids leaving spike data in the
production folder/list.

## 9. Risks

### 9.1 Permission Approval

Even when J is site owner, certain SharePoint operations through Graph
can require admin consent. The spike must handle the case where the
chosen scope is denied and re-record with the next available scope.

### 9.2 Token Audience and Validation

Misconfigured token audiences cause silent 401/403 failures. The spike
should log validation failures explicitly so debugging stays fast.

### 9.3 Production Folder Pollution

The SharePoint round-trip uses real production folder and list paths.
The spike must clean up its test artifacts in the same run. If cleanup
fails, the spike must surface a clear error rather than leave silent
debris.

### 9.4 MSAL Library Surface

MSAL changes can affect bundle size. The spike must measure bundle
size impact and record it in the closure note so v2.4 stays close to
the 370.71 kB baseline.

### 9.5 Dev Mode Drift

If the spike removes `dev-secret-change-me` too aggressively, local
development can break. Q5 keeps Option A until C5 closure. C5 closure
flips to Option B.
