# v2.5 C13 Inline Actions Lite Micro-spec

Date: 2026-05-16
Status: DRAFT
Parent release: v2.5
Predecessor sub-task: C12 Workflow Queue
Sub-task: C13 Inline Actions Lite
Follow-up sub-task: C14 Workflow Action History or C13.1 Polish

## 1. Status

This micro-spec defines C13, the first write-enabled follow-up to the
C12 Workflow Queue.

C12 proved that the app can show a read-only daily collector queue by
merging:

```text
Committed snapshot party rollup
  + SharePoint workflow metadata
  -> /workflow queue
```

C13 adds a small set of inline actions to that queue. The goal is not to
turn the queue into a full editor. The goal is to let the collector make
the most common small updates without leaving the queue.

## 2. Direction

C13 follows this principle:

```text
Small workflow writes, not workflow automation.
```

The queue remains calm and accounting-focused. Actions should feel like
lightweight bookkeeping, not an alert/task automation system.

### 2.1 Goal

Allow a collector to update workflow metadata directly from `/workflow`
for a single party/currency/direction row.

The first action set is:

- mark contacted,
- snooze / next follow-up date,
- quick memo append or replace,
- promise status update.

### 2.2 Why Lite

C9/C10/C11 already provide richer workflow entry points from the
Workbench and Party Detail. C13 should not duplicate every field or
create a second full editor.

C13 Lite:

- reuses existing SharePoint workflow row shape,
- reuses existing C8 write path,
- updates one row at a time,
- avoids bulk operations,
- avoids automatic email sending,
- avoids notifications,
- avoids multi-user conflict handling beyond basic last-write behavior.

## 3. Decision Recap

| Question | Decision | Note |
|---|---|---|
| Q1 Scope | Inline Actions Lite | Small row-level writes only. |
| Q2 Surface | `/workflow` row expansion or compact action panel | Do not add a separate page. |
| Q3 Write path | Reuse C8 SharePoint workflow list write path | No new SharePoint list. |
| Q4 Actions | Contacted, Snooze, Memo, Promise Status | Keep owner edit out unless already simple. |
| Q5 Data model | Existing fields first, one optional timestamp field only if needed | Avoid schema growth. |
| Q6 Conflict behavior | Basic last-write wins with visible updatedAt | Full conflict resolution deferred. |
| Q7 Verification | Source invariants + browser smoke | C12 pattern. |
| Q8 Release shape | v2.5.1 or v2.6 depending size | Do not disturb v2.5.0 tag. |

## 4. Product Purpose

The daily collector opens `/workflow`, reviews the queue, and makes
small status updates:

1. I contacted this party today.
2. Follow up again on this date.
3. Add a short note.
4. Mark a promise as Open, FollowUp, Kept, Broken, or Settled.

The queue should remain a daily work surface. It should not become a
full CRM, payment system, automatic email sender, or task scheduler.

## 5. Included Actions

### 5.1 Mark Contacted

Purpose:

```text
Record that the collector contacted the party today.
```

Recommended storage:

Use existing memo field at first if no dedicated field exists:

```text
memoText += "[YYYY-MM-DD] Contacted."
updatedAt = now
updatedBy = current signed-in user
```

Preferred future storage:

```text
lastContactedAt
lastContactedBy
```

But new SharePoint columns are optional and should not block C13 Lite.

### 5.2 Snooze / Next Follow-up Date

Purpose:

```text
Move the row out of immediate attention until a selected date.
```

Storage option A, no new column:

```text
promiseDate = selected follow-up date
promiseStatus = FollowUp
```

Storage option B, future:

```text
nextFollowUpDate
```

C13 Lite should prefer option A unless J explicitly adds a new
SharePoint column.

### 5.3 Quick Memo

Purpose:

```text
Add short context without opening the full Party Detail workflow editor.
```

Recommended behavior:

- Small textarea in row expansion.
- Save writes `memoText`.
- Optional append mode can be added later.
- Preserve line breaks.
- Show "Saved to SharePoint" on success.

### 5.4 Promise Status Update

Purpose:

```text
Change the workflow state of an existing promise.
```

Allowed values:

```text
None
Open
FollowUp
Kept
Broken
Settled
```

C13 should not create a new promise amount/date editor unless the row
already has promise fields visible from C10. Full promise editing stays
in the existing Owner / Memo / Promise panel.

## 6. Explicitly Out of Scope

C13 does not include:

- bulk update,
- automatic email sending,
- reminder notification,
- background polling,
- Teams notification,
- Outlook send,
- Graph Mail.Send,
- owner picker,
- owner identity normalization,
- multi-user conflict resolution,
- row locking,
- audit-history list,
- new dashboard analytics,
- K1 alias normalization.

If a change requires new SharePoint columns or a new list, it should be
split into C13.1 or C14.

## 7. Data Model

C13 writes to the same workflow row keyed by:

```text
workspaceId + partyKey + currency + direction
```

Required existing fields:

```text
workspaceId
workflowKey
partyKey
partyName
currency
direction
ownerDisplayName
memoText
promiseDate
promiseAmount
promiseStatus
updatedAt
updatedBy
```

Optional future fields:

```text
lastContactedAt
lastContactedBy
nextFollowUpDate
lastAction
```

C13 Lite should avoid optional fields unless J confirms the SharePoint
list schema has been updated.

## 8. API Design

### 8.1 Existing Write Path

C13 should reuse the C8 workflow metadata write route if it already
exists.

Expected shape:

```text
POST or PUT /api/workflow/item
Authorization: Bearer <Microsoft delegated token>
Content-Type: application/json
```

Payload:

```json
{
  "workspaceId": "...",
  "workflowKey": "...",
  "partyKey": "...",
  "partyName": "...",
  "currency": "CAD",
  "direction": "receivable",
  "patch": {
    "memoText": "...",
    "promiseStatus": "FollowUp",
    "promiseDate": "2026-05-20"
  }
}
```

### 8.2 Response

The API should return the updated workflow row:

```json
{
  "status": "ok",
  "source": "sharepoint-workflow-update",
  "item": {
    "workflowKey": "...",
    "memoText": "...",
    "promiseStatus": "FollowUp",
    "updatedAt": "..."
  }
}
```

The frontend should update the queue row from the returned item.

### 8.3 Error Handling

On failure:

- keep the row expanded,
- keep the user's typed text,
- show a clear inline error,
- do not silently discard edits,
- do not update the local queue as if the save succeeded.

## 9. UI Design

### 9.1 Location

C13 actions live inside `/workflow`.

Recommended surface:

```text
Row expansion panel below the queue row
```

Reason:

- Keeps table compact.
- Avoids crowding action buttons into every row.
- Reuses the expanded-row mental model from C2.

### 9.2 Row Action Button

Existing C12 row action:

```text
Open Party
```

C13 may add:

```text
Actions
```

or may make the row expandable with a chevron.

Avoid adding more than two visible action buttons in the base row.

### 9.3 Action Panel Contents

Suggested layout:

```text
Inline Actions

[Mark contacted today] [Snooze date input] [Set FollowUp]

Memo
[textarea]
[Save memo]

Promise status
[select: None/Open/FollowUp/Kept/Broken/Settled]
[Save status]
```

### 9.4 Visual Tone

Use subdued colors.

Avoid:

- red alert banners for normal follow-up,
- animated urgency,
- large colored badges,
- modal-heavy interaction.

Use:

- small status text,
- restrained borders,
- clear disabled/loading states,
- "Saved to SharePoint" confirmation.

## 10. Filter Interaction

After an inline action:

- If the row no longer matches the current filter, it may disappear.
- The UI must make that understandable.

Example:

```text
Saved. Row no longer matches current filters.
```

For C13 Lite, it is acceptable to leave the row visible until refresh if
that avoids confusing row jumps. The implementation must choose one
behavior and keep it consistent.

Recommended:

```text
Recompute filters immediately, but show a short saved message.
```

## 11. Standing Rules Maintained

C13 must preserve all standing rules:

- USD and CAD remain separated.
- AR and AP are never netted.
- Workflow writes are scoped to one `workflowKey`.
- Browser-local snapshot remains a valid fallback.
- SharePoint metadata is additive to the committed snapshot.
- Email drafts are not automatically sent.
- The queue remains calm, not alert-driven.

## 13. Risks

### 13.1 Dual Write Surfaces

Owner/Memo/Promise already has write surfaces in the Workbench. C13 adds
another write surface from the Queue.

Mitigation:

- Keep C13 limited.
- Reuse the same API and row shape.
- Show returned SharePoint row after save.

### 13.2 Filter Disappearing Row

After saving status or date, a row may no longer match current filters.

Mitigation:

- Show saved message.
- Recompute consistently.
- Avoid silent disappearance if possible.

### 13.3 Free-text Owner Identity

`Owner: Me` is still imperfect while owner is free text.

Mitigation:

- Do not make owner picker part of C13 Lite.
- Leave owner normalization for a future sub-task.

### 13.4 SharePoint Latency

Inline saves can feel slow if each action waits for Graph.

Mitigation:

- Use per-row saving states.
- Disable only the affected action.
- Do not block the whole queue.

### 13.5 Scope Creep

Inline actions can easily expand into CRM, reminders, and email
automation.

Mitigation:

- Keep C13 Lite.
- Split larger write features into C14 or later.

## 14. Success Criteria

C13 is successful when:

- `/workflow` remains usable as a queue.
- A row can be expanded.
- At least two small inline actions persist to SharePoint.
- The updated metadata survives refresh.
- C12 read-only baseline remains conceptually intact except for the
  explicit action panel.
- No bulk operation or automatic send behavior is introduced.
- C12 and C13 invariants pass.
- Build passes.
- Browser smoke passes.

Minimum acceptable C13 Lite:

```text
Quick memo save + promise status update
```

Stretch goal:

```text
Mark contacted + snooze
```

## 15. Implementation Order

Recommended order:
