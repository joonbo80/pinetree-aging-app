# v2.5 C12 Workflow Queue Micro-spec

Date: 2026-05-13
Status: DRAFT
Parent release: v2.4.0
Predecessor sub-tasks: C5 / C6 / C7 / C8 / C9 / C9.1 / C10 / C11
Sub-task: C12 Workflow Queue (read-only)
Follow-up sub-task: C13 Inline Actions (out of scope here)

## 1. Status

This micro-spec defines C12, the first v2.5 sub-task.

v2.4 added SharePoint-backed workflow metadata: owner assignment,
memo, promise tracking, email draft history, and row indicators. The
data is now stored, but the daily collector has no single view that
answers the question "what should I work on today, in what order".

C12 introduces that view. It is a read-only daily work queue that
merges the committed snapshot party rollup with SharePoint workflow
metadata and surfaces it as a sortable, filterable queue.

C12 is intentionally read-only. Inline actions (mark contacted, snooze,
quick memo edit) belong to C13 and are explicitly out of scope here.

## 2. Direction

v2.5 prioritizes presenting v2.4 workflow data as a daily work queue
before adding more data-entry surfaces.

### 2.1 Goal

Provide a single screen where the authenticated collector can see, in
priority order:

- which parties have owner, memo, promise, or email-draft context,
- which promises are due today, this week, or already past,
- which parties belong to them as owner versus to others,
- which currency and direction each row represents.

### 2.2 Why Read-Only First

C9/C10/C11 already provide data-entry paths from the Workbench and
Party Detail. Duplicating those entry paths inside the queue would
fragment the source of truth and create two UIs to keep in sync.

Read-only queue:

- reuses C2 selector aggregations,
- reuses C8 list query,
- avoids dual write paths,
- proves daily-queue value before adding inline actions.

### 2.3 Not Yet

C12 does not include:

- inline owner change,
- inline memo edit,
- inline promise update,
- inline email draft creation,
- mark contacted / snooze actions,
- bulk operations,
- notifications.

Those belong to C13 or later.

## 4. Product Purpose

The daily collector should be able to open the queue and answer:

1. What is on my list today?
2. What promise is due today or already overdue?
3. Which parties have memo or recent email draft context?
4. Which parties do not yet have an owner?

The queue must remain a calm accounting workbench, not an alert
dashboard. Visual cues stay subdued. The screen should be readable for
eight hours without fatigue.

## 5. Data Model

### 5.1 Inputs

C12 merges two existing data streams. Both are already produced by
v2.3 and v2.4.

Input A: committed snapshot party rollup
- Source: `selectAgingReport()` and underlying selector helpers.
- Provides: partyKey, partyName, currency, direction, open balance,
  overdue amount, due/aging bucket, signals (Statement Diff, 90+,
  Duplicate, Missing Due), and existing C2 badges.

Input B: SharePoint workflow metadata
- Source: `PE Aging&Statement Workflow` SharePoint list, fetched
  through `aging-api`.
- Provides: workflowKey, partyKey, currency, direction,
  ownerDisplayName, memoText, promiseDate, promiseAmount,
  promiseStatus, updatedAt, updatedBy.
- Row indicators (C9.1) and email draft history (C11) may be linked
  by the same workflowKey.

### 5.2 Merge Key

Each queue row corresponds to a unique combination:

```text
workflowKey = (partyKey, currency, direction)
```

This matches the C2 party rollup grouping and the C8 list schema.

AR/AP separation and USD/CAD separation must be preserved at the queue
row level. A queue row never represents a netted or cross-currency
aggregate.

### 5.3 Derived Fields

Each queue row derives the following fields for sorting and filtering:

- `dueBucket`: `today`, `tomorrow`, `thisWeek`, `overdue`, `later`,
  `noPromise`.
- `isMine`: true when `ownerDisplayName` matches current signed-in user.
- `isUnassigned`: true when no owner is set.
- `hasMemo`: true when `memoText` is present and non-empty.
- `hasOpenPromise`: true when `promiseDate` is set and
  `promiseStatus` is not `Settled`.
- `lastEmailDraftAt`: optional, from C11 history if available.

`dueBucket` is computed from `promiseDate` versus today's date in the
user's local timezone.

### 5.4 Sort Order Default

The default queue order is:

1. `isMine` desc (mine first).
2. `dueBucket` order: overdue, today, tomorrow, thisWeek, later,
   noPromise.
3. open balance desc within the same bucket.

This is the multi-dimension order chosen in Q2.

## 6. Filters

C12 exposes a small filter set. Filter chips follow the v2.3 C2 quick
filter pattern so that the visual language is consistent.

Filters:

- Owner: `Me`, `Unassigned`, `All`.
- Currency: `USD`, `CAD`, `All`.
- Direction: `AR`, `AP`, `All`.
- Due bucket: `Overdue`, `Today`, `This week`, `Has promise`,
  `No promise`, `All`.
- Has memo: yes / no / any.
- Status: from `promiseStatus`. Defaults exclude `Settled` rows.

Defaults on first open:

```text
Owner: All
Currency: All
Direction: All
Due bucket: All
Has memo: any
Status: not Settled
```

Filter state is URL-encoded the same way C2 does:

```text
/workflow?owner=me&due=overdue,today
```

Clear All resets to the defaults above.

## 7. UI Design

### 7.1 Route

New route:

```text
/workflow
```

Reachable from:

- Dashboard, via a new tile `Open Workflow Queue`.
- Direct hard-load with snapshot restored from C7.

`/workflow` must not appear when the workspace has no committed
snapshot. The same "No data loaded yet" empty state used by `/aging`
applies.

### 7.2 Layout

Three regions, top to bottom:

1. Header strip
   - Route title: `Workflow Queue`.
   - asOfDate display, reused from C2.
   - Source pill: `Server` or `Local Snapshot`, reused from C7.

2. Filter strip
   - Filter chips listed in Section 6.
   - Clear all filters control.
   - Result count `Showing X of Y rows`.

3. Queue table
   - One row per workflowKey.
   - Columns defined in Section 7.3.

No tabs. The whole queue is a single ordered list. Grouping happens
through the sort order, not separate tabs.

### 7.3 Columns

Suggested column order:

1. Owner (display name or "Unassigned").
2. Party (partyName, with currency and direction badge).
3. Open balance (signed, currency-aware).
4. Promise (date + amount + status), or "-" when none.
5. Memo (truncated preview, hover or row-click expands).
6. Last activity (most recent of `updatedAt`, `lastEmailDraftAt`).
7. Row indicators (C9.1 visual hints).

The Party column doubles as the row link. Clicking it navigates to
the corresponding `/aging` row in C2 Workbench or to Party Detail,
following the existing v2.3 navigation conventions.

### 7.4 Row Link Behavior

Clicking the Party column navigates to:

- Party Detail page when one exists for that party.
- Otherwise the C2 Workbench filtered to that party.

No editing happens inside the queue row in C12. The row click only
navigates.

### 7.5 Empty States

The queue has three distinct empty states:

1. No committed snapshot. Show the same "Load Baseline" path as
   `/aging`.
2. Snapshot present, but no workflow metadata rows. Show:
   `No workflow data yet. Open the Statement Collection Workbench to
   add owner, memo, or promise.`
3. Snapshot and workflow data present, but the current filters match
   zero rows. Show:
   `No rows match the current filters.`
   Plus a Clear all filters control.

### 7.6 Calm Visual Rules

C12 must preserve the v2.3 visual identity:

- No flashing badges.
- No alert colors as primary surface.
- Status colors used only where they help scanning.
- High-density rows are acceptable, but not crowded.
- USD/CAD numbers always separated visually.
- AR and AP rows never share a single row.

## 8. Data Sourcing and Freshness

### 8.1 Snapshot Layer

The committed snapshot remains the source of truth for the party
rollup, balances, and aging buckets. This matches Q5: snapshot stays
committed-time, workflow stays fresh.

When the user reloads the queue:

- The snapshot does not refetch by itself.
- Workflow metadata refetches.
- The snapshot is only refreshed when the user reimports or restores.

### 8.2 Workflow Metadata Layer

The queue fetches workflow rows from `aging-api` using the C8 path.

Initial call:

```text
GET /api/workflow/list
```

Expected response shape (informal):

```text
{
  rows: [
    {
      workflowKey, partyKey, currency, direction,
      ownerUserId, ownerDisplayName,
      memoText,
      promiseDate, promiseAmount, promiseCurrency, promiseStatus,
      updatedAt, updatedBy
    },
    ...
  ],
  fetchedAt: ISO8601
}
```

The fetch is server-first. If the call fails:

- Fall back to the last cached workflow snapshot in `localStorage`
  if one exists.
- Show a small `Stale` indicator and the last `fetchedAt`.
- Never silently merge stale workflow rows without showing the
  staleness.

### 8.3 Merge Behavior

For each unique `workflowKey`:

- If snapshot has the row and workflow has the row, merge.
- If snapshot has the row but no workflow row, the queue still shows
  the row with empty Owner/Memo/Promise.
- If workflow has a row but snapshot does not have the matching party
  rollup row, the queue must not display that orphan workflow row.
  Instead, surface an orphan count in a small badge in the header
  strip, similar to how `NOT_IN_ERP_EXTRACT` is surfaced in the
  Review Queue.

Orphan workflow rows are a Review Queue concern, not a queue row.

## 9. Standing Rules Maintained

C12 must preserve all standing rules from v2.3 and v2.4:

- USD and CAD must not be merged.
- AR and AP must not be netted at any aggregation layer.
- `NOT_IN_ERP_EXTRACT` remains a Review Queue workflow concern.
- Browser-local snapshot remains a valid fallback.
- The Workbench identity stays a calm accounting workbench, not an
  alert dashboard.
- Audit-driven decisions: every queue row's data source is identifiable
  (Snapshot or Workflow Metadata).

## 10. Out of Scope

C12 explicitly does not include:

- Inline owner change.
- Inline memo edit.
- Inline promise update.
- Inline email draft creation.
- Mark contacted, snooze, or other status updates.
- Bulk row operations.
- Notifications, reminders, or background polling.
- Real-time multi-user editing.
- New SharePoint columns.
- New workflow metadata fields beyond those already in the C8 schema.
- K1 alias normalization (still WATCHLIST).

### 11.2 Backend

No new backend endpoints are required if the existing C8 list query is
sufficient. If a dedicated queue endpoint is preferred, add:

- `GET /api/workflow/queue?asOf=YYYY-MM-DD`

That endpoint should reuse the same SharePoint list path and return
the same row shape. Server-side filtering is optional in C12.

### 11.3 Caching

The workflow fetch result should be cached to `localStorage` under:

```text
agingApp.workflowQueueCache.v1
```

Cache invalidation: every successful fetch overwrites the cache, and
the cache is read only when the live fetch fails.

## 13. Risks

### 13.1 Selector Cost

Merging snapshot rollups with workflow metadata can be slow if the
selector recomputes on every render. The implementation should memoize
the merge by `(snapshotImportBatchId, workflowFetchedAt, filters)`.

### 13.2 Owner Identity Match

`isMine` depends on a stable comparison between the signed-in user and
`ownerDisplayName` or `ownerUserId`. If the C9 schema uses display
name, the queue must use the same field. Mismatched fields can make
the queue show empty for the right user.

### 13.3 Orphan Workflow Rows

Workflow rows can exist for parties no longer present in the current
snapshot (party renamed, removed, or split). The queue must not
silently hide them. Surface a small orphan count badge instead.

### 13.4 Workflow Cache Staleness

Stale cache can show incorrect promises or memos. The Stale indicator
in Section 8.2 must be honest and visible.

### 13.5 Scope Creep

C12 is read-only. Pull requests that introduce inline actions belong
to C13, not C12. Reviewers should reject inline-action drift inside the
C12 scope.

### 13.6 Visual Drift

Queue density and color use must match the calm-workbench style.
Reviewers should reject alert-style colors and bouncing badges.

## 14. Success Criteria

C12 is complete when:

- `/workflow` route loads with snapshot present.
- Queue merges snapshot rollup with SharePoint workflow rows.
- Default sort and filter match Sections 5.4 and 6.
- Row click navigates correctly.
- Empty states render appropriately.
- Stale indicator displays when workflow fetch fails.
- Automated invariants suite passes.
- Browser smoke passes.
- All v2.3 and v2.4 release-gate suites stay green.
- No inline actions are introduced.
