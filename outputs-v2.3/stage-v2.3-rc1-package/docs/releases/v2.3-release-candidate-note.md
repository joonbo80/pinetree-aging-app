## 2. Product Definition

v2.3 C2 is the read-only decision layer of a small collection system.

It answers the daily collector question:

> Today, who needs follow-up, why, in which currency, and for how much?

The product identity established in v2.3:

> Calm accounting workbench, not alert dashboard.

This release intentionally does not add persistent collection actions,
owner assignment, memo storage, payment-promise tracking, or SharePoint
workflow persistence. Those belong to Phase 3.

### 3.3 Accounting Safety Rules

v2.3 locks these accounting rules:

- USD and CAD are never combined.
- AR and AP are not silently netted.
- Party rollups group by partyKey + currency + direction.
- Tab totals preserve USD/CAD and AR/AP separation.
- CSV export preserves party-rollup direction separation.
- Overdue bucket totals preserve USD/CAD and AR/AP separation.
- NOT_IN_ERP_EXTRACT rows stay in Review Queue workflow; they are not
  mixed into Aging Workbench party rollup export.
- No Due Date rows are not silently assigned to Current or Overdue.

## 5. Fixes Applied During RC Verification

### 5.1 API Version Sync

Finding:

- `/api/health` returned version 1.2.0 while `aging-api/package.json`
  declared 1.2.1.

Fix:

- `aging-api/src/routes/health.ts` now reports `SERVICE_VERSION = '1.2.1'`.
- `tools/test-c4-polish.mjs` now verifies API health service version
  matches `aging-api/package.json`.
- C4 expected count updated from 15/0 to 16/0 in F1 docs and invariants.

### 5.2 F1 No Due Date Wording

Finding:

- F1 Scenario 4 originally asked for USD/CAD and AR/AP separation in
  the No Due Date callout.
- Actual implementation shows USD/CAD totals in the callout.
- Baseline open No Due Date rows are receivable-only, so AR/AP split is
  validated in tab totals and Aging Breakdown instead.

Fix:

- F1 Scenario 4 now expects USD/CAD totals in the No Due Date callout
  and notes where AR/AP separation is validated.

### 5.3 Files Tab Size Fallback

Finding:

- Baseline API file rows can omit `sizeBytes`, causing the Files tab to
  render `NaN KB`.
- The same file used mojibake fallback dashes.

Fix:

- `aging-app/src/components/tabs/FilesTab.tsx` now formats missing file
  size as `-`.
- fallback dash rendering is ASCII-clean.

## 6. Known Deferrals

### 6.1 C3 Snapshot Import/Export

Still pending. This remains the next large functional sub-task.

Purpose:

- Persist and restore imported workbench state without reloading the
  baseline flow.
- Prepare for Phase 3 persistence patterns.

### 6.2 K1 Alias Normalization

Conditional. The known LEMOND alias issue remains non-blocking for v2.3
release candidate, because party alias normalization belongs to the
v2.4/v3 data-linking area.

Known example:

- LEMOND transactions use `lemond-food-corp-new-addr`.
- Some statement rows use `lemond-food`.
- Exact partyKey matching can split one business across aliases.

### 6.3 Direct Deep Link Without Committed State

The `/aging` route reads the current committed workspace state. A fresh
hard load of `/aging` without a committed in-memory result shows the
intended "No data loaded yet" state.

This is expected before C3 Snapshot persistence and Phase 3 SharePoint
persistence. The verified user path is:

```text
Upload + Parsing Preview -> Load Baseline Demo -> Confirm Import
-> Dashboard -> Open Collection Workbench
```
