## 2. Product Definition

C2 is the read-only decision layer of a small collection system.

It is designed to answer the daily collector question:

> Today, who needs follow-up, why, in which currency, and for how much?

C2 is not just an aging report. It is also not a CRM, payment system, or persistent collection action log. Those workflows require SharePoint persistence and belong to Phase 3.

C2 gives the accounting team a calm workbench for statement and aging review:

- Which parties are current, overdue, cleared, or missing due dates.
- Which rows have statement differences.
- Which rows are 90+ days.
- Whether the balance is AR or AP.
- Whether the currency is USD or CAD.
- Which rows should be exported for follow-up.

The design philosophy established during C2:

> Calm accounting workbench, not alert dashboard.

This means:

- Reduced saturation.
- Thin left rails instead of loud alert panels.
- Subtle row tints.
- Compact badges.
- High scan value for long daily use.

## 4. Locked Domain Rules

### 4.1 AR/AP Separation

AR and AP must never be silently netted.

This rule is enforced at every C2 aggregation layer:

- Party rollup rows.
- Tab totals.
- CSV export rows.
- Overdue bucket breakdown.

If the same party has both receivable and payable rows, those rows remain separate.

### 4.2 Currency Separation

USD and CAD must never be combined.

C2 does not perform currency conversion. Any future conversion feature needs a separately reviewed FX policy and audit trail.

### 4.3 Workflow Separation

C2 is party-rollup driven.

`NOT_IN_ERP_EXTRACT` statement rows are not included in C2 party rollups or C2 CSV export because they do not have a matched ERP transaction. They belong to the Review Queue workflow.

Future work can add a Review Queue export, but that should be a separate workflow export.

### 4.4 No Due Date Handling

No Due Date items are not silently classified into Current or Overdue.

Open No Due Date rows are shown in a pinned callout:

- Open No Due Date: 118

Settled No Due Date rows remain in Cleared:

- Settled No Due Date: 185

This avoids burying a review-required accounting condition inside a normal tab.

### 4.5 Cleared Precedence

Cleared classification has precedence over date rules.

If `isZeroBalance === true`, the transaction belongs to Cleared even if a due date is missing or old.

## 6. Architecture

### 6.1 Pure Helper Pattern

C2 uses pure helpers for domain logic:

| File | Responsibility |
|---|---|
| `aging-app/src/selectors/agingReport.ts` | Main workbench selector |
| `aging-app/src/components/aging/agingFilterLogic.ts` | Multi-AND tab filtering |
| `aging-app/src/components/aging/exportCsv.ts` | CSV generation |
| `aging-app/src/components/aging/agingBucketLogic.ts` | Overdue bucket breakdown |

Invariant tests import production helpers directly. This avoids the tautological test pattern where a test re-implements production logic and only proves itself.

### 6.2 URL-Backed Filters

Round 5 made filter state shareable:

- `asOf`
- `currency`
- `direction`
- `signal`

This is intentionally not localStorage. URLs are more useful for accounting collaboration because a view can be copied and reviewed by another user.

### 6.3 Component Structure

```text
AgingReportPage
  NoDueDateCallout
  AgingFilters
  AgingExportButton
  AgingTabs
    AgingBucketBreakdown (Overdue only)
    PartyRollupTable
      TransactionRows
```

### 6.4 CSV Export

CSV export format:

- UTF-8 with BOM.
- RFC 4180 escaping.
- Party rollup level.
- 15 columns.
- `Export filtered` and `Export all`.
- Blob download with deferred object URL revoke.

Filename examples:

```text
aging-2026-05-01-all.csv
aging-2026-05-01-CAD.csv
aging-2026-05-01-CAD-AR-statementDiff.csv
```

## 9. Standing Rules Established

1. Production helpers should be pure where possible.
2. Invariants should import production helpers directly.
3. Do not re-implement production logic inside invariant tests when direct import is possible.
4. AR/AP separation applies at every aggregation layer.
5. USD/CAD separation is mandatory unless a reviewed FX policy exists.
6. Workflow exports should respect workflow boundaries.
7. URL-backed filters are preferred for shareable accounting review states.
8. CSV exports must use UTF-8 BOM for Excel compatibility.
9. Blob download object URLs must be revoked after a delay, not synchronously.
10. UI should stay calm and scan-friendly for daily accounting work.
11. Baseline facts should be pinned by invariant tests, not prose alone.

## 10. Silent-Corruption Lessons From C2

C2 added several important failure-pattern lessons:

| Pattern | Defense |
|---|---|
| Wrong asOfDate path in selector | Pre-invariant baseline simulation |
| Tautological invariant re-implements logic | Import production helper directly |
| AR/AP netting in party rollups | Include `direction` in aggregation key |
| Node runtime pass but TypeScript build fail | Run build after invariant suite |
| Reference workspace differs from mainline | Inspect mainline before patching |
| Workflow wording hides routing decision | Separate Review Queue orphan workflow |
| CSV export could mix workflows | Exclude NotInErpExtract from C2 CSV |
| Browser object URL revoked too early | Deferred revoke pattern |
