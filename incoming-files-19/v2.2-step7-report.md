## Architecture additions

### Cross-tab focus (Step 7 mechanism, used by future Steps 8-9 too)

PartyDetailPage now owns a focusedTxId state. The Statements tab calls
onFocusTransaction(txId) which:

1. Sets focusedTxId in parent
2. Switches activeTab to 'transactions'

The Transactions tab observes focusedTxId and:

1. Bypasses the Direction filter for the focused row (so settled tx
   targeted from a SETTLED_AFTER_STATEMENT statement link is still
   visible even though OPEN default would normally hide it)
2. Scrolls the focused row into view
3. Applies tx-row-focused class which fires a 1.8s background flash
4. Clears the focus state via setTimeout

This pattern will be reused by:

- Reviews tab (Step 8): click a review item that references a tx
- Duplicates tab (Step 9): click a duplicate group member

So the same focus mechanism is the foundation for all cross-tab
navigation in v2.2.

### Match type priority (reviewer P2 corrected)

```js
function matchTypePriority(link, isStrict) {
  if (link.matchType === 'NOT_IN_ERP_EXTRACT' && isStrict) return 1;
  if (link.matchType === 'BALANCE_DIFFERENCE')             return 2;
  if (link.matchType === 'CHANGED_AFTER_STATEMENT')        return 3;
  if (link.matchType === 'NOT_IN_ERP_EXTRACT')             return 4;  // broad
  if (link.matchType === 'SETTLED_AFTER_STATEMENT')        return 5;
  if (link.matchType === 'EXACT_SIGNED')                   return 6;
  return 7;  // edge types
}
```

The 'isStrict' parameter is computed from the party's review items —
a statement link is "strict" if the same (sourceFile, sourceRow) pair
appears in a NOT_IN_ERP_EXTRACT review item. This is the link from the
strict 7-row Dashboard count to the per-statement-link view.

### Match filter taxonomy (reviewer P1 #2 corrected)

Old: All / Matched / Unmatched / Has Difference  (which conflated
matched-but-different with matched, and was confusing)

New: All / Exact / Matched with difference / Unmatched (outside extract) /
Settled after statement

Each option maps to one or more matchType values. Definitions are
explicit and non-overlapping.

### Label distinction (reviewer P1 #1 corrected)

Same matchType code (NOT_IN_ERP_EXTRACT) renders differently based on
whether a strict review item exists:

- strict: red pill "Confirmed not in ERP"
- broad: yellow pill "Not in uploaded ERP extract"

Footer adds a reconciliation note when broad rows present:

```
"95 statement rows are outside the uploaded ERP extract; 7 are
confirmed review items."
```

This addresses the 95 vs 7 confusion the reviewer flagged.

## Methodology note

Reviewer's mid-spec correction round (4 P1/P2 findings on micro-spec
before code) was the right call. Coding the original micro-spec would
have produced a UI that:

- mislabeled 95 rows as confirmed missing
- used a confusing "Matched" definition
- broken cross-tab focus when target was settled

Each of those would have been a Step 7 review round. Catching them at
the spec stage saved 1-2 review cycles.

This is the second time a micro-spec freeze has paid back in fewer
review rounds (Step 6 had OPEN filter caught at spec stage too, in
hindsight).
