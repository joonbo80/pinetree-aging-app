## Architecture decisions

### Callback prop pattern, not react-router

Dashboard already uses `onOpenReview` / `onOpenReviewCategory` callbacks.
The parent (App.tsx) handles the actual route navigation. Step 12
follows the same pattern with `onOpenParty(partyKey: string)`.

This keeps Dashboard router-free, which was a v2.0 architectural
decision worth preserving.

### partyName -> partyKey fallback map

`TopPartySummary` from Phase 1 v1.3.0 baseline does NOT include
partyKey. Adding partyKey to selector + Phase 1 producer is out of
scope for v2.2 Step 12.

Two-track strategy:
1. Add OPTIONAL `partyKey` to TopPartySummary type (forward-compat)
2. Build a partyName -> partyKey map in Dashboard from
   `result.details.transactions` for legacy payloads

Real-data verification: all 12 baseline topParties resolve via the
fallback map. 0 ambiguous names (each partyName maps 1:1 to one
partyKey across the entire baseline).

When v2.3 (or Phase 1 v1.4.0) starts populating partyKey directly,
the fallback path becomes unused — no breaking change.

### Graceful degrade when key unavailable

```tsx
const canLink = onOpenParty && key !== null;
```

If onOpenParty is not provided OR the partyKey can't be resolved,
the row renders as plain text (current v2.0 behavior). Link upgrades
require BOTH the parent to wire onOpenParty AND the partyKey to be
resolvable.

This is defensive: a payload with topParties but no details (rare
schema variant) won't crash — it just won't link.

## Reviewer focus areas

1. **Type extension is additive** — TopPartySummary.partyKey is
   optional, so existing v2.0/v2.1 payloads still typecheck.

2. **Fallback map is defensive** — when partyKey is missing AND
   nameToKey lookup fails, the row degrades to plain text, not
   crashes.

3. **Aria-label on link button** — `Open party detail for {name}`
   gives screen readers the action verb + context.

4. **No react-router import in Dashboard** — Step 12 maintains the
   v2.0 callback-prop architecture.

5. **App.tsx wiring is one line** — `onOpenParty={(k) => navigate(`/party/${k}`)}`.
   See apply-order.md for the exact snippet.

## Open question for reviewer

The fallback map currently builds on every render of Dashboard. For
12 topParties + 1230 transactions this is trivially fast (~0.5ms),
but if real production data scales to 10,000+ transactions, this
could be useMemo'd. Cosmetic — happy to add if reviewer prefers.

## Methodology note

Step 12 was the smallest code step in v2.2 (4 files, ~70 lines net).
The biggest decision was *which* architecture to use for the
fallback (partyKey-direct vs fallback map). I chose the additive
approach to keep selector / Phase 1 producer untouched.

This is the third time v2.2 has hit a "small step, big architectural
implication" pattern:
- Step 6 OPEN filter (UX default vs spec freeze)
- Step 11 file split (csvExport.ts as first separation)
- Step 12 partyKey extension (forward-compat type vs forced refactor)

Each time, the choice that preserved the most existing surface area
turned out to be correct. v2.3 onward should expect the same — small
architectural choices add up.
